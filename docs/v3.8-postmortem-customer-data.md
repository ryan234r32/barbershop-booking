## 2. 8 個 root cause + 防範措施

### RC1: Vercel function fire-and-forget kill
**症狀**: webhook 第一次 push 不出 Flex，要點第二次才會出。
**根因**: `lineClient.pushMessage().catch(...)` 不 await → Vercel 回傳 response 後 freeze function → dangling promise 被殺。
**已修**: PR #48，改 await。
**防範**:
- ✅ Code 已有註解標出這個 pattern (`bookings/route.ts:281-283`)
- 📌 **新規則**: 任何 Vercel function 內部不要寫 `.catch()` fire-and-forget。**必 await**。或用 Next.js 16 的 `after()` API。
- 📌 加進 CLAUDE.md「Landmines」段。

### RC2: Admin manual booking 永遠造 ghost user
**症狀**: 同一個客人 (e.g. Ryan) 在後台搜尋出現 N 筆「新客 1 次」。
**根因**: POST `/api/bookings` 對 admin path 永遠合成 `manual-{adminId}-{uuid}` → 每次都建新 User。
**已修**: PR #49，admin path 接受 `customerId` → 直接 link 既有 user。
**防範**:
- 📌 任何「為使用者建 record」的 API path，必須有「使用既有 record」的選項
- 📌 frontend「自動填入」UX 必須真的把選擇 propagate 到 API（不能只填 placeholder text）

### RC3: 客戶搜尋無防呆
**症狀**: 打 1 個字就 query；無 debounce；DB 全表掃描。
**根因**: `if (customerName.length < 1)` 邊界錯；300ms debounce 太短；`displayName` 沒 index。
**已修**: PR #51，min 2 字 + 500ms debounce。
**防範**:
- 📌 定義一組「搜尋輸入框」標準 hook：min 字數 + debounce + min query length
- 📌 Phase 3 (DB index) 等 Phase 2 後實測決定

### RC4: Tenant 隔離不一致
**症狀**: `/api/customers/[id]` GET + PATCH 沒檢查 tenantId。
**根因**: pattern 在 booking 端點都套了，customer 漏掉。
**已修**: PR #51 + 6 個自動測試。
**防範**:
- 📌 Code review checklist：「所有讀寫 user-owned record 的 API 是否都有 `tenantId: admin.tenantId`」
- 📌 Prior learning `v1.3-hardening-missed-GET-read-paths` 已涵蓋這類事，但只 grep `POST` 不夠 → 必須 audit ALL HTTP methods
- 📌 將來 V4.0 multi-tenant SaaS 上線前 → 全 codebase 跑一次「all admin endpoints 是否都 enforce tenant」audit

### RC5: dedup script 重算 totalVisits 用錯演算法
**症狀**: Ryan totalVisits 16 → 9（少 7）。
**根因**: 我寫的腳本用 `count(status: "COMPLETED")` 重算，但系統其他地方是 `increment: 1` 累計（在 `/checkout` 和 `/[id]` PATCH）。**兩者語意不同**。
**已修**: `merge-chen-and-cleanup.ts` 改成 `legacy.totalVisits + canonical.totalVisits` (sum)。
**防範**:
- 📌 **不要重算 counter 欄位**。如果系統用 `increment` 累計 → 合併時 sum，不要 recompute。
- 📌 看到一個欄位之前先 grep 看它 *如何* 被寫入（不只看怎麼讀）

### RC6: dedup script 把追蹤訊息寫到 user-visible 欄位
**症狀**: 老闆點預約詳情看到「[merged-from-ghost dc36e88c-3240-... on 2026-04-29]」x9 行的「亂碼」。
**根因**: 我寫腳本時把 trace note 直接 prepend 到 `user.notes`，但這個欄位是給 admin 寫客戶筆記用的，會在預約詳情頁顯示。
**已修**: 清理掉所有以 `[merged-from-ghost ` 開頭的行。
**防範**:
- 📌 **任何 user-visible field 都不該被腳本當作 audit log 用**。
- 📌 應該另開 `UserAuditLog` table 或寫到 `console.log` + 報告檔案。
- 📌 「腳本要不要寫資料」這個決策前，先看一下這個欄位**會不會出現在 UI**（grep "notes" in components）。

### RC7: clean-test-data.ts 太貪心，連真實 prod booking 都刪
**症狀**: Phase 2 dedup 完隔天，Ryan 的 15 筆 booking 只剩 2 筆 CONFIRMED（兩筆都是 4/30 新建的）。
**根因**: `clean-test-data.ts` 邏輯是「刪 ID 不是 hist- 開頭的所有 booking」。Ryan 的真實 LIFF / 手動預約沒有 hist- 前綴 → 全部刪光。
- 這個 script 原本是 V3.5 demo 前用來「清測試資料保留 Excel 歷史」的一次性工具
- 但留在 repo 裡 + 預設 dry-run，老闆或別的 process 容易誤觸 `--commit`
**已修**: 一次性損失，bookings 復原困難。
**防範**:
- 📌 把 `scripts/clean-test-data.ts` rename 成 `archived/` 並加大警告 banner
- 📌 任何破壞性 script 都該有 expiry：跑完一次後改名 / 移到 archive 資料夾
- 📌 CLAUDE.md「Landmines」加一行：**有 `--commit` flag 的 script 必須有 git tag / migration 紀錄**

### RC8: 「同一個客人」分散在 legacy + LIFF 兩筆 record
**症狀**: 老闆後台搜尋「陳昶龍」 vs 「陳昶龍 Ryan」 出現 2 個獨立 record，但其實是同一個人。
**根因**: 系統有 LIFF profile 自動 link 機制（`/api/profile/me` 用電話比對 legacy user），但 legacy 沒填電話 → link 從沒觸發。
**已修**: 手動 merge script。
**防範**:
- 📌 LIFF profile 頁加「綁定既有客戶資料」按鈕：客人填了電話 → 顯示「我們有找到一筆過去的紀錄（4 次來訪），是您嗎？」→ 同意 → 觸發合併
- 📌 Excel 匯入時要求至少有一個聯絡欄位（電話 / email）才匯入

## 4. 給未來自己的清單

下次要動 production user 資料前，**先回答這 6 個問題**：

1. ☐ 我要動的欄位**怎麼被寫入**？（grep `field_name.*=`、`field_name.*update`、`field_name.*increment`）
2. ☐ 我要動的欄位**會不會出現在 UI**？（grep `field_name` 在 components/）
3. ☐ 我要刪的資料 **能不能 cascade 到別人**？（schema.prisma 看 FK + onDelete）
4. ☐ 我的腳本 **--dry-run 跑出來的 report 老闆能看懂嗎**？
5. ☐ 動的範圍 **能不能限縮**（`--limit N` 先驗證一小批）？
6. ☐ 跑完後 **怎麼驗收正確性**（post-flight audit）？
