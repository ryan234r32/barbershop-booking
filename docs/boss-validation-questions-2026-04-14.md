## YC 提問心法（帶進會議）

1. **Specificity is the only currency** — 不要接受「大概」「都可以」。要逼出具體數字 / 人名 / 場景。
2. **Watch, don't demo** — 每問一題就讓他**實際點手機**，看他手指猶豫在哪。
3. **Status quo is the real competitor** — 永遠先問「你以前怎麼處理？」再問「這樣取代 OK 嗎？」
4. **Interest ≠ demand** — 老闆說「好啊」不算數，要問「如果這個功能明天消失，你會 upset 嗎？」
5. **Push twice** — 第一個答案通常是客套話，追問第二次才是真心話。

**紀錄技巧**：每題留兩個欄位 — ✅ 他的回答 / 🚩 我原本的規則。兩者不一致 = 要改。

## 🟦 A. 營業時段與時段邏輯（Booking Availability Engine）

**我目前的規則：**
- 營業時間 **11:00–20:00**（硬編碼 default）
- 時段單位 **1 小時**
- 每天 **9 個時段**
- 剪髮=1 格、燙=3 格、染=4 格
- 最多可連 8 格
- `startHour + slotsNeeded ≤ closeHour` 強制驗證（不能 19:00 預約 3 小時燙髮）
- 過去日期一律拒絕
- Holiday 列表內的日期整天無法預約

### 要問的問題

1. **[開放]** 「你實際上每天幾點開、幾點關？是不是固定？」→ 驗證 11:00-20:00 是不是真的
2. **[封閉]** 「有沒有中午休息時間？午休要不要不開放預約？」→ 目前沒中斷
3. **[三選一]** 「一個時段是：A 60 分鐘 / B 30 分鐘 / C 依服務而定」→ 目前寫死 60
4. **[封閉]** 「禮拜幾公休？每週固定嗎？還是不定？」→ 目前 businessHours 表支援但要驗
5. **[開放]** 「你最後一個客人最晚幾點開始做？」→ 如果染髮要 4 小時，19:00 根本不能收
6. **[封閉]** 「客人可以提前多久預約？3 天？2 週？1 個月？」→ **目前沒限制！** 客人可以預約半年後，這是漏洞
7. **[封閉]** 「客人可以多晚預約當天？例如下午 2 點能預約當天 3 點嗎？」→ 目前可以，沒有前置時間
8. **[三選一]** 「同一天同一個客人：A 只能預約一次 / B 可以預約多次 / C 不設限」→ 目前 C
9. **[封閉]** 「國定假日你會開嗎？還是當公休？」→ 目前要人工加 holiday
10. **[開放]** 「如果臨時要休一天（例如生病），你現在會怎麼處理？」→ 驗證 holiday 管理 UX
11. **[封閉]** 「已經有客人預約的日子如果要臨休，系統要幫你自動通知他們嗎？」→ 目前沒做

## 🟦 C. 取消政策（Cancellation Policy）

**我目前的規則：**
- **≥24 小時前** → 線上免費取消，不記違規
- **<24 小時 + 營業時間內** → 必須打電話（APP 擋住，不算違規）
- **<24 小時 + 非營業時間** → 只顯示「請營業時間來電」
- **No-show（沒來）才算違規**
- **取消永遠不算違規**（打電話取消也不算）

### 要問的問題

18. **[開放]** 「客人臨時取消對你的傷害多大？一個月會損失多少營收？」→ 衡量政策嚴格度
19. **[三選一]** 「取消的緩衝時間要設：A 12 小時 / B 24 小時 / C 48 小時」→ 目前 24
20. **[封閉]** 「<24h 取消真的要逼他打電話嗎？還是線上取消就好（但記違規）？」→ 這是核心決策
21. **[三選一]** 「<24h 取消的客人：A 算違規 / B 不算違規（你決定面對面處理）/ C 看理由」→ 目前 B
22. **[封閉]** 「取消的時候要不要請客人填原因？你會想看嗎？」→ 目前沒收集
23. **[開放]** 「VIP 客人臨時取消，你會特別放水嗎？」→ 要不要做 VIP 豁免
24. **[封閉]** 「同一個客人一個月取消幾次你會擔心？」→ 目前沒「取消次數」監控
25. **[三選一]** 「取消後要不要收費：A 不收 / B 收一部分訂金 / C 不收但記次數」→ 目前 A，沒訂金機制

## 🟦 D. 改期流程（Reschedule）

**我目前的規則：**
- 改期**沒有次數限制**
- 改期**沒有時間限制**（前一小時都能改）

- 改期**取消舊的提醒通知，排新的**

### 要問的問題

## 🟦 F. 金流 / 付款流程（Payment）

**我目前的規則：**
- **只收現金 / 銀行轉帳**，沒有線上金流
- 轉帳要**上傳截圖**到 Supabase Storage
- Payment 狀態：`PENDING → RECEIVED / WAIVED`
- 現金預設 method=CASH，由 past-due modal 標記收款
- 轉帳需要**老闆手動確認**才變 RECEIVED
- **沒有付款期限**（客人可以到店前才付）

### 要問的問題

43. **[開放]** 「現在客人付現 vs 轉帳，實際上比例大概多少？」→ 決定要不要優化轉帳流程

45. **[封閉]** 「轉帳截圖你會仔細看嗎？還是大概瞄一下？」→ 判斷自動對帳是否有價值
