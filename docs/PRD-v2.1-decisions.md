> **產出日期**：2026-04-15
> **用途**：逐條跟老闆再確認後，併入 PRD V2.1

## 新增需求

以下為訪談中老闆提出、目前 PRD 未涵蓋的項目，建議納入 V2.1 scope。

### 1. 預約開放窗口：45 天

PRD 未定義預約可預訂多遠的未來。老闆明確收斂為 45 天。

> 「原本想『半年』，討論後收斂為 **45 天**」
> 「老闆自己會在 1 個月前才排私人行程（例如小孩暑假出國），太久會跟客戶衝突」

- 45 天內 → 系統直接開放預約
- 45 天外 → 不顯示時段；特殊案例走「專人聯絡」

> 「少數國外客會約 2–3 個月後 → 走『專人聯絡』流程，系統不直接放行事曆」

### 2. 漂髮諮詢待回覆流程

PRD 的預約流程是「選服務 → 選日期 → 選時段 → 確認」，假設所有服務都能自助完成。老闆指出漂髮無法這樣做。

> 「**漂**：客戶不一定知道漂完要不要染、想要什麼顏色 → **這一塊一定要人腦判斷**」
> 「把可以勾選的拆成 A/B/C/D，真的拆不了的（= 漂髮諮詢）就進『諮詢待回覆』流程」

- 可自助的服務：剪、燙、染、補染 → 走現有預約流程
- 不可自助的：漂髮、漂+染 → 進入「諮詢待回覆」佇列，老闆手動回覆
- Admin 端需要一個「待處理諮詢」區塊，浮在最上面

### 3. 訊息分級收件匣（Admin 端）

PRD 只有預約管理，沒有訊息/通知分級的概念。老闆最大痛點是忙碌時無法及時回覆 LINE。

> 「讓這一筆預約能夠更快確認，避免兩人溝通時間太久」
> 「**把『系統能解決的』自動化，『系統無法解決的』浮到最上面**」
> 「45 天外（特殊、漂髮諮詢類）→ 浮在最上面需要親自回覆」
> 「系統能解決的沉到下面，需要出馬的浮到最上面」

- 上層：需要老闆親自處理（漂髮諮詢、特殊預約、異常情況）
- 下層：系統已自動處理（一般預約確認，按「已知道」即可）

### 4. 對帳 / 收款簡化 UI

PRD 的付款流程是「現金到店付 / 轉帳截圖上傳」，但沒有 Admin 端的快速收款 UI。

> 「理想：預約結束前 20 分鐘，App 自動跳出『本次多少錢，顧客XXX』→ 按『現金』或『已轉帳』→ 完成」

- 服務結束前 20 分鐘自動推送收款卡片
- 一鍵選擇「現金」或「已轉帳」
- 取代目前 Excel 手動輸入流程

### 5. 轉帳末五碼對帳（過渡方案）

PRD 寫「手動確認轉帳」，但沒有具體流程。老闆提出更清楚的過渡方案。

> 「顧客在 LIFF 按轉帳 → 自動跳出銀行帳號 → 顧客提供末五碼 → 老闆後台對帳核對」

- LIFF 付款頁顯示銀行帳號 + 引導輸入轉帳末五碼
- Admin 後台可用末五碼快速比對銀行 App 通知

### 6. 支出 / 損益追蹤

PRD 只有營收統計，完全沒有支出面。老闆明確要求。

> 「老闆希望系統能自動算收入，**也要能輸入支出** → 看『收入 - 支出 = 真實獲利』」
> 「固定支出（租金、水電）vs 變動支出（耗材、補貨） → 分兩類處理」

- 新增 Expense model：type（FIXED / VARIABLE）、amount、category、date、note
- 儀表板新增「損益」區塊：收入 - 支出 = 淨利

### 7. 常見問答自動回覆（LINE Bot）

PRD 的 Webhook 只處理 follow/unfollow/message（回覆引導訊息）。老闆希望關鍵字觸發結構化問答。

> 「客戶 LINE 打到『燙』、『染』、『漂』這些關鍵字 → 自動跳方塊」
> 「方塊點下去 → 『請提供 A. 目前髮況照片 B. 想要的成品照片 C. XXX』」
> 「節省老闆每次重複打字」

- 關鍵字觸發 Flex Message 回覆模板
- 至少支援：燙、染、漂 三組
- 可在 Admin 設定自定義關鍵字 + 回覆內容

### 8. 行事曆 UI 改進

PRD 有行事曆功能但沒有詳細 UI spec。老闆看完原型後的反饋。

> 「行事曆時間格『想更窄一點』，希望一眼看到一整天」
> 「行事曆要能縮放（類似 Google Maps 手勢）」

- 時間格高度縮小，一螢幕顯示完整 11:00-20:00
- 支援 pinch-to-zoom 手勢（平板為主要裝置）

### 9. Excel 歷史數據匯入

PRD 沒有資料匯入功能。老闆有 2-3 個月的 Excel 營業紀錄。

> 「請老闆提供近 2-3 個月的 Excel 做匯入，讓報表立刻有歷史數據」

- 需要 Admin 端的 CSV/Excel 匯入功能
- 欄位對應：新男剪、新女剪、新學童剪、童剪、新染、補染、燙、瀏海、漂、護髮、西髮、廠商、其他
- 區分現金（黑字）vs 轉帳（紅字）

## 決策 Pending

以下為訪談內容與 PRD 現有規劃有衝突，需要明確決策的項目。

### 1. CRM 分群門檻不一致

**PRD 現有**：
- REGULAR = 1-5 次來訪 + 60 天內有活動
- VIP = ≥ 6 次來訪

**訪談說法**：
- REGULAR = 60 天內 + **6 次以上**
- VIP = 60 天內 + **12 次以上**

> 「NEW → REGULAR（60 天內 + 6 次以上）→ VIP（60 天內 + 12 次以上）」

**建議**：以訪談為準（老闆的實際判斷），更新門檻為 REGULAR ≥ 6 次、VIP ≥ 12 次。同時確認 AT_RISK 門檻——PRD 寫 60-120 天，訪談寫 100 天。

> 「AT_RISK（100 天沒來）→ LAPSED（半年沒來）」

### 2. 取消政策簡化

**PRD 現有**：複雜的三路判斷（明天以後 / 當天營業時間 / 當天非營業時間），含違規記錄。

**訪談說法**：只分兩種。

> 「**24h 以上 → 線上直接取消**」
> 「**24h 內 → 必須打電話**（增加取消成本）」
> 「點取消時引導『要不要改期』，很多情況會因此變成改期而非取消」

**建議**：簡化為老闆版本（24h 分界），取消時優先引導改期。PRD 的違規計次機制是否保留需確認——訪談沒有提到違規懲罰。

### 3. 改期政策：從「2h 前限制」改為「全開放」

**PRD 現有**：V1.2 有改期功能但未明確定義時間限制（代碼中可能有 2h 限制）。

**訪談說法**：

> 「老闆想了之後：**全部開放線上改**（前一小時也可以）」
> 「客戶都會提前 1 天以上改；當天很少，且當天有事通常會 LINE 溝通」
> 「先全開放，若有人真的很誇張再針對性加規則（95–98% 客戶都 OK）」

**建議**：移除時間限制，改期完全開放。

### 4. 「新客」定義：店面制 vs LINE 帳號制

**PRD 現有**：NEW = 0 次來訪（基於系統內預約紀錄）。

**訪談說法**：

> 「老闆的『新客』= 這間店面的新顧客（換店後計算）」
> 「不是 LINE 帳號的新註冊」
> 「**需要支援『搬店後的舊客 → 新客分類』**」

**建議**：User model 加 `isNewToCurrentLocation` flag，或基於店面搬遷日期（3 年前）來判斷。匯入的 Excel 歷史客戶標記為「舊店客戶」。需確認具體判斷邏輯。

### 5. 金流串接意願

**PRD 現有**：明確寫「不做線上金流」（設計決策）。

**訪談說法**：

> 「金流串接（1% 手續費）：老闆覺得『**1% 可以接受**』」

**建議**：V2.1 先做末五碼過渡方案（見新增需求 #5），金流串接列入 V2.2+ 評估。不急著翻轉 PRD 的設計決策，但記錄老闆已同意成本。

### 6. 儀表板時間維度

**PRD V2.1 規劃**：日/週/月營收折線圖。

**訪談說法**：

> 「**季度視角**（不是月，月太短看不出趨勢）」
> 「新客 vs 舊客比例 / 流失率」
> 「月營收、週營收、年營收」
> 「時段占用率（例：100 個時段中 14 個被預約 → 14%）」

**建議**：保留日/週/月，新增季度與年度視角。加入「時段占用率」指標（PRD 現有的 KPI 只看預約完成率，沒有占用率）。

### 7. 公休日設定

**PRD 現有**：每天個別設定是否營業（靈活）。

**訪談說法**：

> 「**公休**：禮拜三、禮拜日（之前一度說只有禮拜日，更正為『禮拜三 + 禮拜日』）」

**建議**：功能不衝突，但需確認是否為固定公休（每週三、日都休）還是保持靈活設定。目前系統支援每天個別設定，seed 資料設為週三、日不營業即可。

## 移除項目

以下為老闆明確否決或不需要的功能，應從 V2.1 roadmap 排除。

### 1. 半年預約窗口

原本討論過讓客戶預約半年後的時段，老闆否決。

> 「原本想『半年』，討論後收斂為 **45 天**」

收斂為 45 天，超過走專人聯絡。

### 2. 排班偏好 / 智慧集中排班

PRD V2.0 有設計師排班，且 V1 有智慧時段推薦（傾向集中排班）。老闆表示不需要刻意集中。

> 「**排班偏好**：沒有特別偏好，不用刻意集中或分散」

智慧推薦的「緊接在現有預約之後 +10 分」邏輯可保留（減少空檔），但不需要做為強制排班策略。此項不影響現有代碼，僅記錄老闆態度。

### 3. 半小時時段

PRD 已決策為「1 小時最小單位」，訪談再次確認。

> 「**時段單位**：1 小時（不做半小時）」

維持現有設計，不需重新評估。
