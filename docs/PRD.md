## 1. 產品概述

### 1.1 願景

為台灣小型理髮廳打造一套**以 LINE 為核心的預約管理系統**，取代「手動 LINE 訊息 + Excel」的低效管理方式，讓店家能自動化處理預約、通知、客戶關係，並讓顧客在 LINE 內完成所有操作。

### 1.2 產品定位

**先單店驗證，再逐步擴展為多店 SaaS 平台。**

| 階段 | 定位 | 目標 |
|------|------|------|
| V1 | 單店專屬系統 | 為第一家試驗店上線，驗證產品可行性 |
| V2-V3 | 功能完善 | 加入設計師排班、行銷、評價等進階功能 |
| V4 | 多店 SaaS | 開放自助開店、訂閱制，服務多家店 |

### 1.3 目標用戶

**店家端（管理者）：**

| 店家規模 | 優先級 | 備註 |
|----------|--------|------|
| 1 人店（老闆兼設計師） | V1（首發） | 不需設計師排班，最簡單的場景 |
| 2-3 人小型店 | V2 | 需要基本設計師指派 |
| 4+ 人中型店 | V2+ | 需要完整排班、班表、業績管理 |

**顧客端：**
- 台灣的 LINE 使用者（LINE 在台灣滲透率 > 90%）
- 透過 LINE LIFF（LINE Front-end Framework）操作，無需下載額外 App

## 2. 核心問題與解決方案

### 2.1 店家痛點

| 痛點 | 現況 | 影響 |
|------|------|------|
| 預約管理混亂 | 用 LINE 私訊接預約，再手動記到 Excel | 容易漏單、重複預約、忘記通知 |
| 客戶追蹤困難 | 無法知道誰多久沒來、誰是 VIP | 無法做精準行銷，客戶流失無感 |
| 取消/爽約無約束 | 口頭約定，顧客隨意取消 | 空檔浪費，營收損失 |
| 營運數據空白 | 無任何統計報表 | 無法做經營決策 |

### 2.2 解決方案

| 痛點 | 解決方案 |
|------|----------|
| 預約混亂 | LINE LIFF 自助預約 + 自動衝突檢查 + Redis 鎖防重複 |
| 客戶追蹤 | 自動 CRM 分群（NEW → REGULAR → VIP → AT_RISK → LAPSED） |
| 取消/爽約 | 分層取消政策 + 違規計次 + 自動限制機制 |
| 數據空白 | Admin 儀表板 + 營收/客流/服務分析 |

## 3. 使用者角色

### 3.1 顧客（LINE 用戶）

- **入口**：LINE 官方帳號 → LIFF 頁面
- **身份驗證**：LINE LIFF SDK 自動取得 LINE User ID
- **可執行操作**：
  - 選擇服務 → 選日期 → 選時段 → 確認預約
  - 檢視自己的預約記錄
  - 線上取消預約（依取消政策）
  - 上傳轉帳截圖作為付款證明
- **接收通知**：預約確認、24 小時提醒、1 小時提醒、取消確認

### 3.2 店家管理員（Owner）

- **入口**：`/login` → Admin Dashboard
- **身份驗證**：Email + Password → JWT（httpOnly Cookie，7 天有效）
- **可執行操作**：
  - 查看今日排程與營運統計
  - 管理所有預約（新增、完成、標記爽約、管理員取消）
  - 管理客戶資料（搜尋、篩選、查看詳情）
  - 管理服務項目（新增、編輯、停用）
  - 設定營業時間、假日、銀行轉帳資訊
  - 查看數據分析報表

### 3.3 店家員工（Staff）— V2

- 與 Owner 相似但權限受限
- 只能查看/操作自己的預約
- 不能修改營業設定

## 4. 功能規格 — V1

> 圖例：✅ 已完成 ｜ 🔧 需調整 ｜ 📋 需補完

### 4.1 顧客端 — LIFF 預約介面

#### F-C01: 預約流程 ✅

**5 步驟預約流程：**

```
Step 1: ServiceStep（選擇服務）
  → 顯示所有啟用中的服務，含名稱、描述、時長、價格
  → 選擇後進入下一步

Step 2: CalendarStep（選擇日期）
  → 日曆元件，禁用過去日期與假日
  → 選擇後自動載入該日可用時段

Step 3: TimeStep（選擇時段）
  → 顯示所有可用時段（1 小時為單位）
  → 智慧推薦標記（⭐ Recommended）
  → 多時段服務自動檢查連續空檔

Step 4: ConfirmStep（確認預約）
  → 顯示完整預約摘要（服務、日期、時間、價格）
  → 可輸入備註
  → 確認送出

Step 5: SuccessStep（預約成功）
  → 顯示預約編號
  → 提供「查看我的預約」與「繼續預約」按鈕
```

**技術細節：**
- Redis 分散式鎖（TTL 10 秒）防止同時段重複預約
- 預約建立時自動排程 24h + 1h LINE 提醒通知
- 預約建立時自動發送 LINE Flex Message 確認訊息

#### F-C02: 我的預約 ✅

- 分為「即將到來」（CONFIRMED）和「歷史紀錄」兩個 Tab
- 即將到來的預約可執行取消操作（依取消政策判斷）
- 當天取消會顯示違規警告
- 每筆預約可快速前往付款頁面
- 顯示：服務名稱、日期、時間、狀態、價格

#### F-C03: 付款 ✅

- 支援兩種付款方式：
  - **現金**：顯示「到店付款」提示
  - **銀行轉帳**：顯示店家銀行帳戶資訊 + 截圖上傳功能
- 付款狀態追蹤（PENDING / RECEIVED / WAIVED）
- 已上傳截圖可預覽

### 4.2 管理端 — Admin Dashboard

#### F-A01: 儀表板 ✅

- **統計卡片**：本週預約數、完成數、營收、佔位率
- **今日排程**：時間軸顯示當日所有預約
  - 每筆顯示：時段、顧客名稱、服務名稱、價格
  - 快速操作按鈕：標記完成 / 標記爽約

#### F-A02: 行事曆 ✅

- 週/月視圖切換
- 顯示所有預約在對應時段上
- 點擊預約可查看詳情

#### F-A03: 預約管理 ✅

- **預約列表**：依日期、狀態、顧客篩選
- **新增預約**：管理員手動為顧客建立預約（來源標記為 ADMIN / PHONE / WALK_IN）
- **預約詳情**：查看完整資訊 + 執行狀態操作
- **狀態操作**：
  - `CONFIRMED → COMPLETED`（完成服務）
  - `CONFIRMED → NO_SHOW`（標記爽約）
  - `CONFIRMED → CANCELLED_BY_ADMIN`（管理員取消）

#### F-A04: 客戶管理 ✅

- **客戶列表**：
  - 搜尋（姓名 / 電話）
  - 依分群篩選（NEW / REGULAR / VIP / AT_RISK / LAPSED）
  - 分頁顯示（每頁 20 筆）
  - 顯示：大頭貼、名稱、分群標籤、來訪次數、最後來訪、違規次數
- **客戶詳情**：個人資料、預約歷史、違規紀錄、標籤、備註

#### F-A05: 服務管理 ✅

- 服務 CRUD（新增 / 編輯 / 停用）
- 欄位：名稱、描述、時長（分鐘）、佔用時段數、價格、排序
- 停用的服務不會出現在顧客預約頁面

#### F-A06: 營業設定 ✅

- **商家資訊**：店名、電話、地址、銀行帳戶（轉帳用）
- **營業時間**：每天個別設定開始/結束時間、是否營業
- **假日管理**：新增/刪除特定日期假日（含原因說明）

#### F-A07: 數據分析 ✅

- 預約統計（總數、完成、取消、爽約）
- 營收統計
- 新客戶數
- 客戶分群分佈
- 熱門服務排行
- 每日預約趨勢

### 4.3 系統功能

#### F-S01: LINE Webhook ✅

| 事件 | 處理邏輯 |
|------|----------|
| `follow`（加好友） | 建立 User 記錄 + 發送歡迎 Flex Message（含 LIFF 預約連結） |
| `unfollow`（封鎖） | 記錄日誌 |
| `message`（文字訊息） | 回覆引導訊息，導向 LIFF 預約頁面 |

#### F-S02: 自動提醒通知 ✅

- 預約建立時自動排程兩則通知：
  - **24 小時前**提醒
  - **1 小時前**提醒
- Cron Job 每 **15 分鐘**執行一次，檢查並發送到期通知
- 透過 LINE Push Message 發送 Flex Message
- 預約取消時自動取消對應的待發通知
- 每批次最多處理 50 則

#### F-S03: CRM 自動分群 ✅

| 分群 | 條件 |
|------|------|
| `NEW` | 0 次來訪 |
| `REGULAR` | 1-5 次來訪，最近 60 天內有活動 |
| `VIP` | ≥ 6 次來訪，最近有活動 |
| `AT_RISK` | > 0 次來訪，60-120 天未來訪 |
| `LAPSED` | > 0 次來訪，> 120 天未來訪 |
| `BLACKLISTED` | 手動標記，不會被自動重新分群 |

- Cron Job **每週一凌晨 4 點**執行重新分群
- VIP 狀態鎖定，不會因間隔降級

#### F-S04: 取消政策與違規追蹤 ✅

（詳見 [5.2 取消政策](#52-取消政策)）

#### F-S05: 智慧時段推薦 ✅

系統會對可用時段進行評分排序，推薦最優時段：

| 規則 | 加分 | 原因 |
|------|------|------|
| 緊接在現有預約之後 | +10 | 減少空檔，整合排程 |
| 緊接在現有預約之前 | +8 | 同上 |
| 越早的時段 | +5 | 前置填滿，減少碎片 |

- 標記前 2 名為「推薦」
- 依推薦優先度排序顯示

#### F-S06: 排程清理 ✅

Cron Job **每天凌晨 3 點**執行：
- 重置已過期的違規限制（超過 1 個月）
- 將過期未到的 CONFIRMED 預約標記為 NO_SHOW
- 清除過期的舊通知紀錄

#### F-S07: 速率限制 ✅

- 中介層（Middleware）對所有 API 請求做速率限制
- 限制：每個 IP 每分鐘 60 次請求
- 記憶體內計數（Vercel 冷啟動時重置）

## 5. 商業規則

### 5.1 營業時間與時段

| 項目 | 規則 |
|------|------|
| 預設營業時間 | 11:00 - 20:00 |
| 時段單位 | 1 小時（"HH:00" 格式） |
| 每日可用時段數 | 9 個（11:00, 12:00, ... 19:00） |
| 每日個別設定 | 支援（每天可設不同開始/結束時間、是否營業） |
| 假日 | 可設定特定日期為假日，該日不可預約 |

### 5.2 取消政策

```
                    ┌──────────────────────────────────────┐
                    │           取消政策判斷流程            │
                    └──────────────────────┬───────────────┘
                                           │
                              ┌─────────────▼─────────────┐
                              │  預約日期是明天或之後嗎？  │
                              └─────────────┬─────────────┘
                                    │               │
                                   YES              NO
                                    │               │
                              ┌─────▼─────┐  ┌──────▼──────┐
                              │ 免費取消  │  │ 是當天預約  │
                              │ 無違規    │  └──────┬──────┘
                              └───────────┘         │
                                           ┌────────▼────────┐
                                           │ 現在是營業時間？ │
                                           │ (11:00-20:00)   │
                                           └────────┬────────┘
                                              │           │
                                             YES          NO
                                              │           │
                                        ┌─────▼─────┐ ┌──▼──────────┐
                                        │ 必須打電話 │ │ 可線上取消  │
                                        │ 不可線上   │ │ 但記違規    │
                                        └───────────┘ └─────────────┘
```

### 5.3 違規制度

| 項目 | 規則 |
|------|------|
| 違規觸發 | 當天非營業時間取消、爽約（NO_SHOW） |
| 違規上限 | 3 次 |
| 限制內容 | 超過 3 次 → 限電話預約 1 個月（線上預約被封鎖） |
| 重置機制 | 限制到期後自動重置違規計數（每日 Cron 檢查） |

### 5.4 服務時段計算

| 服務類型 | 時段數 | 範例 |
|----------|--------|------|
| 剪髮 | 1 slot（1 小時） | 預約 14:00 → 佔用 14:00-15:00 |
| 燙髮 | 3 slots（3 小時） | 預約 14:00 → 佔用 14:00-17:00 |
| 染髮 | 3-4 slots | 預約 14:00 → 佔用 14:00-17:00 或 18:00 |

- 多時段服務必須佔用**連續時段**
- 系統自動檢查連續空檔是否充足

### 5.5 付款

| 項目 | 規則 |
|------|------|
| 付款方式 | 現金（到店付）/ 銀行轉帳（截圖證明） |
| 線上金流 | 不支援（設計決策） |
| 轉帳截圖 | 顧客上傳 → 管理員手動確認 |
| 付款狀態 | PENDING → RECEIVED / WAIVED |

## 6. 資料模型

### 6.1 Entity Relationship 概覽

```
Tenant (店家)
  ├── AdminUser (管理員)         1:N
  ├── User (顧客/LINE用戶)      1:N
  ├── Service (服務項目)         1:N
  ├── BusinessHours (營業時間)   1:N (max 7, per day-of-week)
  ├── Holiday (假日)             1:N
  ├── Booking (預約)             1:N
  │     ├── Payment (付款)       1:1
  │     ├── CancellationRecord   1:1
  │     └── Notification (通知)  1:N
  └── Notification (通知)        1:N
```

### 6.2 關鍵模型摘要

| 模型 | 主要欄位 | 備註 |
|------|----------|------|
| **Tenant** | name, slug, LINE credentials, bank info, plan | 多租戶核心，含 TRIAL/BASIC/STANDARD/PREMIUM 計劃 |
| **AdminUser** | email, password, name, role (OWNER/STAFF) | 每租戶唯一 email |
| **User** | lineUserId, displayName, phone, segment, violationCount | 自動從 LINE 建立，含 CRM 分群 |
| **Service** | name, duration, slotsNeeded, price, sortOrder | slotsNeeded 決定佔用時段數 |
| **Booking** | date, startTime, endTime, status, source | status 有 5 種，source 有 4 種 |
| **Payment** | amount, method, status, screenshotUrl | 1:1 關聯 Booking |
| **CancellationRecord** | isViolation, reason, bookingDate/Time | 取消歷史追蹤 |
| **Notification** | type, scheduledAt, sentAt, status, messagePayload | 排程式通知 |
| **BusinessHours** | dayOfWeek (0-6), startTime, endTime, isOpen | 每天個別設定 |
| **Holiday** | date, reason | 特定日期公休 |

### 6.3 重要索引

- `bookings(tenantId, date, status)` — 查詢特定日期預約
- `bookings(tenantId, date, startTime)` — 時段衝突檢查
- `users(tenantId, segment)` — CRM 分群查詢
- `users(tenantId, lastVisitAt)` — 客戶活躍度排序
- `notifications(status, scheduledAt)` — Cron 查詢待發通知

## 11. 限制與已知取捨

### V1 設計取捨

| 項目 | 決策 | 原因 |
|------|------|------|
| **不做線上金流** | 只支援現金 + 銀行轉帳 | 小店不需要，避免金流成本與法規複雜度 |
| **1 小時最小單位** | 不支援 30 分鐘時段 | 簡化時段計算邏輯，1 人店夠用 |
| **只走 LINE 通知** | 無 SMS、無 Email | LINE 在台灣滲透率夠高，降低成本 |
| **無設計師指派** | 所有時段等價 | V1 目標是 1 人店 |
| **記憶體速率限制** | 非分散式，冷啟動會重置 | V1 規模夠用，V2 可改用 Redis |
| **手動確認轉帳** | 無自動對帳 | 避免銀行 API 整合複雜度 |

### 已知限制

- LINE Bot SDK v10 使用 legacy Client API（非最新的 v2 Messaging API）
- LIFF 必須在 LINE App 內開啟，無法在一般瀏覽器使用
- Vercel Cron 最短間隔 1 分鐘，Free Plan 每天最多 1 次（需 Pro Plan）
- Upstash Redis Free Tier 有每日請求數限制
