# V3.7 — 預約 / 對帳 / 客戶 / 訊息 整合流程設計

> 狀態：**待老闆 confirm，再開工**
> 撰寫：2026-04-29
> 取代：原本散在 daily 對帳頁與 LIFF 預約間的鬆散連動
> 預期影響：減少老闆每日對帳手動工序，提升轉帳對帳信任感

---

## 0. 為什麼要做

V3.6 daily 對帳頁已經把「今日預約對帳清單」做出來，但目前還是**孤立的視圖**，只能改 `settledAt`：
- LIFF 客戶下單 → daily 列表只是被動顯示，**沒有觸發任何 action**
- 老闆從日曆手動建單 → 同樣只是顯示
- 客戶轉完帳回填末五碼 → **沒有 link 到 daily 對帳**，老闆要去訊息頁找
- 老闆按「確認」（settle）→ **客戶不知道**，沒收到「我們已收款」訊息
- 客戶帳號頁面也**看不到自己的末五碼歷史**

把這 4 個破口接起來 = V3.7。

## 1. 老闆原始需求（4/29 訊息）

### 1.1 預約來源整合
- (a) LINE 顧客的預約 + 老闆自己在日曆上建的預約 → 全部 link 到每日對帳單

### 1.2 帳務連動與匯款資訊

## 2. 我的解讀（先 confirm，再做）

我先把每個需求**翻譯成具體 user story**，請你看是不是這個意思。

### Story A — LIFF 預約自動進對帳清單
> 「客戶在 LIFF 下單後，當天的對帳列表就要看得到，不用再去日曆找。」

**現況**：daily view 已經拉所有當日 booking（不分來源）。✅ 已完成。
**需要新做**：在 booking row 上**標記來源**（LIFF / 日曆手建 / 補登），這樣老闆對帳時心裡有底。

### Story B — 老闆日曆手建單也進對帳清單
> 「我自己在日曆排的客戶，也要列在對帳裡。」

**現況**：手建 booking 用 `BookingSource = ADMIN`，已經會出現在 daily list。✅ 已完成。
**需要新做**：同 Story A，加 source badge。

### Story C — 「確認」按鈕觸發 Rich Menu 連動 ⚠️ 需要再 confirm
> 「對帳確認後，要連動到 Rich Menu 的點擊動作。」

我**沒完全理解這條**。可能的解讀：

**解讀 C1**：客戶被「確認對帳」後，他下次按 Rich Menu 時看到「上次消費已對帳」訊息。
**解讀 C2**：對帳確認 = 觸發 Rich Menu 圖案切換（例如把「轉帳資訊」按鈕替換成「下次預約」）。
**解讀 C3**：對帳確認的同時，自動發 Rich Menu shortcut 給客戶（按鈕快速重新預約 / 留評論等）。

→ **請老闆釐清要哪一個**（或都不是）。

### Story D — 末五碼自動 populate 到對帳列
> 「客戶回填末五碼後，那筆預約就標記轉帳已收，daily 對帳列直接顯示末五碼。」

**現況**：客戶在 LIFF 結帳頁回填末五碼 → `Payment.transferLastFive` + `Payment.status = VERIFYING`。
**需要新做**：daily 對帳列加新欄位顯示末五碼，且 row 自動標 ✓ 標記為「待對帳，末五已回」。
老闆比對銀行 app 後按「確認」→ `Payment.status = RECEIVED` + `Booking.settledAt = now`。

### Story E — 末五碼鎖在客戶頁面
> 「客戶頁要能看到他過去每筆轉帳的末五碼歷史。」

**現況**：客戶詳細頁顯示 booking 歷史 + service + price，但**沒拉 Payment 資訊**。
**需要新做**：客戶詳細頁加「付款記錄」段，列出每筆 `Payment.amount + method + transferLastFive + status + receivedAt`。

### Story F — 對帳完發確認訊息給客戶
> 「按『已確認』，自動 push LINE 給客戶說『我們已收到您 NT$X 的款項』。」

**現況**：settle endpoint 只動 `settledAt`，**不發任何訊息**。
**需要新做**：settle 流程加 fire-and-forget LINE push（類似 `sendNewBookingPush` 模式）。
**老闆建立的 booking**（`lineUserId.startsWith("manual-")`）→ 跳過，因為沒真實 LINE 對應。

## 5. 流程圖（end-to-end）

```
客戶 LIFF 預約
  ↓
建立 Booking + 半 Payment (PENDING / 0 元)
  ↓
（如果是轉帳）客戶結帳頁回填末五碼
  ↓ Payment.status = VERIFYING + transferLastFive + verifiedAt
  ↓
[daily 對帳頁 自動顯示]
  ┌─────────────────────────────────────┐
  │ 11:00 王小明 染髮 NT$2,500          │
  │       來源 LINE · 末五 12345        │
  │       [確認] (待對帳)                │
  └─────────────────────────────────────┘
  ↓ 老闆比對銀行 app + 點「確認」
  ↓
PATCH /api/bookings/:id/settle
  ↓
1. Booking.settledAt = now
2. Booking.status = COMPLETED (若 CONFIRMED → auto)
3. Payment.status = RECEIVED, receivedAt = now
4. Fire-and-forget LINE push: 「您 NT$2,500 已收款 ✓」
  ↓
客戶手機收到 LINE 訊息
  ↓
客戶詳細頁「付款記錄」section
  顯示這筆 + 末五 + 收款時間
```

## 6. 風險 / Edge cases

| Edge case | 處理 |
|---|---|
| 老闆手建 booking（`manual-*` userId） | settle 時跳過 LINE push（沒真實對應） |
| 現金付款（沒末五碼） | row 不顯示末五欄位；settle 同樣 push |
| Walk-in 補登（沒 user 連結） | 同上跳過 push |
| settle → push 失敗（LINE rate limit / 網路） | log error，不阻塞 settle 成功 |
| 客戶被 push 確認訊息 → 又重新 settle 一次 | settle 是 idempotent，第二次 short-circuit 不重發 push |
| 客戶在 LIFF 看到「已收款」訊息但帳務糾紛 | 對帳 DELETE endpoint 可 unsettle，但**不會自動撤回 LINE 訊息**（LINE API 限制）— 老闆需手動發更正訊息 |
