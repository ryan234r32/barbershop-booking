V2 基於以下新發現，對 V1 做出本質性的調整：

1. **真實用戶痛點確認** — 透過老闆訪談，確認最大痛點是「**時間**」（回訊息 + Excel 對照），不是「漏單」
2. **使用場景重新定位** — 老闆在**剪頭髮空檔用手機**管理預約，Admin 必須 **mobile-first**
3. **競品研究完成** — 確認訂閱制為業界標準，定價區間 NT$300（夯客）～ NT$12,000（Mangomint）
4. **客人端體驗重構** — 從 5 步驟 wizard 改為**單頁三區**，學習 Fresha 的流暢度
5. **Onboarding 重新定位** — 從「設定導覽」改為「體驗導覽」（設定由開發者在訪談時完成）
6. **商業模式定義** — 精品管理服務路線，NT$5,000-8,000/月

## 1. 產品概述

### 1.1 願景

為台灣小型理髮廳（1-3 人店）打造一套 **LINE 為核心、手機優先**的預約管理系統，讓店家老闆能在剪頭髮空檔用手機輕鬆管理所有預約，讓顧客不離開 LINE 就能完成預約、改期、付款。

### 1.2 產品定位

**精品管理服務路線。** 不走自助 SaaS、不打價格戰、不做通用型。針對 1-3 人小型精品店，提供客製化體驗 + 專屬支援。

| 階段 | 定位 | 狀態 |
|------|------|------|
| **V1** (已上線) | 1008 Hair Studio 單店驗證 | ✅ 已完成 |
| **V2** (本文件) | 基於實際使用優化 UX + 商業模式建立 | 🚧 規劃中 |
| V3+ | 第 2-3 家店 onboarding，驗證可複製性 | ⏳ 未來 |
| V4+ | 開放多店 SaaS（如果前面驗證成功）| ⏳ 未來 |

### 1.3 目標用戶

**店家端（V2 聚焦）：**
- **1008 Hair Studio**（1 人店，老闆兼設計師）
- 營業時間：11:00-20:00
- 每日預約量：6-8 人（約 60-80% 滿）
- 付款方式：現金 + 銀行轉帳（手動確認截圖）

**顧客端：**
- 台灣 LINE 使用者（LINE 滲透率 > 90%）
- 年齡：20-50 歲，熟悉 LINE 操作
- 使用場景：躺在沙發、通勤路上、任何有 LINE 的時刻

### 1.4 V2 核心設計原則

1. **手機優先** — 老闆和客人的主要操作介面都是手機
2. **時間節省優先** — 每個設計決策都問「這讓老闆省了多少時間？」
3. **設計先行** — 每個新頁面先做 HTML mockup，確認後才寫 code
4. **學習經過驗證的 UX patterns** — 不重新發明輪子，學 Fresha / 夯客已驗證的模式
5. **客製化 > 彈性** — 1008 不需要無限選項，需要「專為他打造」的體驗

## 2. 核心問題與解決方案

### 2.1 店家痛點（基於老闆訪談）

| 痛點 | 老闆原話 | V1 狀態 | V2 解決方案 |
|------|---------|---------|------------|
| **時間被訊息吃掉** | 「我花很多時間在弄，回覆訊息、排安排」 | 部分解決（LIFF 預約）| Admin 手機版讓老闆 5 秒看完今日 |
| **手機 + Excel 切換麻煩** | 「同一隻手機看就轉去很麻煩」 | 已解決（有 admin 後台）| Admin mobile-first 讓手機操作更順 |
| **想陪家人但被訊息綁住** | 「我可以跟小孩陪玩、陪看書」 | 部分解決 | 減少不必要通知、一鍵標記完成 |
| **客人等回覆** | 「更快回復體驗，不用那麼久」 | 部分解決（自動確認）| LIFF 單頁預約，客人 30 秒完成 |
| **臨時取消難管理** | 「臨時單篇取消真的會比較麻煩」 | 部分解決（違規機制）| V2 新增：取消釋出通知（waitlist）|

### 2.2 顧客痛點（推論）

| 痛點 | V1 狀態 | V2 解決方案 |
|------|---------|------------|
| 預約步驟太多（5 步驟）| 有 | 單頁三區（服務 + 日期時段 + 備註）|
| 選完日期才看時段，不順手 | 有 | 日曆與時段同頁顯示 |
| 不知道哪個時段老闆方便 | 有 | 智慧推薦時段（已有 smart-suggest.ts，V2 上線）|
| 想改期只能取消重訂 | 有 | V2 新增：改期功能 |
| 時段全滿很無奈 | 有 | V2 新增：加入 waitlist，有空自動通知 |

### 2.3 V2 的解決方案總覽

| 痛點類別 | 解決方案 |
|---------|---------|
| LIFF 客人端 | 單頁預約流程、日曆+時段同頁、智慧推薦、改期、waitlist |
| Admin 老闆端 | 手機優先、今日時間軸、一鍵操作、報表一眼看完 |
| LINE 訊息 | Flex Message 美感優化、品牌統一 |
| 體驗引導 | 體驗型 Onboarding（非設定型）|
| 商業模式 | 訂閱制 + 30 天試用 + 精品支援 |

## 3. 使用者角色與場景

### 3.1 顧客（LINE 用戶）

**身份驗證：** LINE LIFF SDK 自動取得 LINE User ID，無需註冊

**典型使用場景：**

| 場景 | 何時發生 | 在哪裡 | 需要什麼 |
|------|---------|--------|---------|
| 發現想剪頭髮 | 下班、週末前 | 沙發、通勤 | 快速預約，不想切 App |
| 想改期 | 臨時有事 | 任何地方 | 一鍵改期，不想被懲罰 |
| 查看預約 | 到店前 | 路上 | 看時間、地址、付款資訊 |
| 收到提醒 | 預約前一天 | 任何地方 | 確認 / 改期 / 取消 |
| 完成付款 | 剪完之後 | 店裡 | 現金或轉帳截圖 |

**可執行操作：**
- 瀏覽服務（含照片、說明、時長、價格）

- 加入 waitlist（V2 新增）
- 上傳轉帳截圖
- 接收 LINE 通知（確認 / 提醒 / 取消 / waitlist 釋出）

### 3.2 店家老闆（1008 Hair Studio Owner）

**身份驗證：** Email + Password（JWT httpOnly Cookie）

**典型使用場景：**

| 場景 | 何時發生 | 在哪裡 | 需要什麼 |
|------|---------|--------|---------|

| 客人打電話來要改期 | 隨時 | 店內或家裡 | 手機快速改時段 |
| 晚上結帳 | 打烊前 | 店內 | 看今日總收入 |
| 週一公休 | 家裡 | 沙發 | 看本週統計、下週預約 |
| 新客預約 | 收到推播 | 任何地方 | 看客人資料、做準備 |

**可執行操作：**
- 查看今日時間軸（V2 主畫面）
- 查看週 / 月行事曆
- 標記預約狀態（完成 / 爽約 / 取消）

**技術細節：**

#### 月曆式日期選擇器

- 7 欄網格顯示整個月（取代橫向 date strip）
- 選中日期以深森林綠圓形 highlight
- 預設顯示當月，可切換上/下月
- 「今天」按鈕快速跳回當月
- **日期狀態（由 backend 計算）：**
  - 可預約：正常顯示，可點擊
  - 不可預約：直接灰掉 disabled（客人點不到）
    - 包含：過去日期、休息日（週一）、假日、超過預約窗口、**該服務當日無任何可行連續空檔**
  - 今天：特殊 highlight（深森林綠底線或小點）
  - 選中：深森林綠圓形實心 + 暖乳白文字

#### 時段選擇邏輯（做法 A：每個可行整點起點）

**後端計算規則（`src/lib/booking/availability.ts` 已實作）：**
- 輸入：服務時長 N 小時、目標日期、該日已預約時段、老闆營業時間
- 輸出：所有可行的開始時段列表
- 條件：`開始時間 + N 小時 ≤ 營業結束` 且 `開始到結束之間所有時段都是空的`

**顯示範例（染髮 3 小時，4/21 營業 11:00-20:00，14:00 已被預約）：**

```
上午
[11:00] 推薦     ← 11:00-14:00，剛好接到現有預約，無空檔浪費

下午
[15:00]         ← 15:00-18:00
[16:00]         ← 16:00-19:00
[17:00]         ← 17:00-20:00
```

（12:00 和 13:00 不顯示，因為會跨越 14:00 的已預約）
（18:00 和 19:00 不顯示，因為會超出 20:00 結束時間）

**顯示規則：**
- 格子**只顯示開始時間**（例：`[11:00]`），不顯示範圍
- 時段按上午 / 下午分組
- 已滿時段保留顯示為 disabled（讓客人感覺店家搶手）
- 超出營業時間的時段不顯示（避免混淆）
- 選中時段後，下方顯示 **Hint Box**：
  ```
  💡 預約時長：180 分鐘
  你的預約時間：11:00 — 14:00
  ```

#### 智慧推薦邏輯（`smart-suggest.ts`）

**推薦優先順序：**
1. **優先：能無縫接續現有預約**的時段（減少零碎時間）
2. **次優：最早的可用時段**（填前面的空檔）

**老闆訪談依據：**
> 「今天三點有空、五點也有空。我會先跟客人說三點。因為五點可以吃個飯，或者後面如果又有約就直接接上。」

推薦的時段會有「推薦」badge（深森林綠框 + 小標籤）。客人仍可自由選擇其他時段。

#### 多時段服務的月曆 Highlight（新增）

當客人選了一個多時段服務（例如染髮 3 小時）+ 選了某個時段（例如 11:00）：
- **時段區：** 11:00 顯示為選中（實心深森林綠）
- **月曆區：** 維持不變（選中日期 21 日還是原本的圓形 highlight）
- **Hint Box：** 顯示「你的預約時間：11:00 — 14:00」

**重點：月曆只標記「選中哪一天」，時段區只標記「從哪一點開始」，Hint Box 顯示完整範圍。**

> 備註：時段格子本身的視覺語意是「開始時間」，不是「佔用時間」。若後續發現客人混淆，可考慮在時段區以較淺的色塊標示「將被佔用」的後續時段，但 V2 先不做。

#### 特殊情境處理

**客人選了一個不可預約的日期（例如該服務當日無連續空檔）：**
- 理想狀態：月曆上該日期應該直接灰掉，客人根本點不到
- 保險機制：如果因為某些原因客人點到了（例如 race condition），時段區顯示：
  ```
  此日期 [服務名] 已滿
  最近可預約：4/23 (四)  [立即查看]
  ```
- 「立即查看」按鈕會自動切到最近的可預約日期

#### 其他技術細節

- 單頁捲動，不跳轉
- 日期/時段同頁顯示（點日期 → 時段區自動 fetch 更新，有 skeleton loading）
- Sticky 底部確認列（永遠可見）
- 確認按鈕觸發 → 若首次預約 → 跳出資料收集 bottom sheet（見 F-C01a）
- 確認後 inline 顯示成功狀態，不跳頁
- Redis 分散式鎖（TTL 10 秒）防止重複預約

#### F-C01a: 首次預約資料收集 🆕

**觸發時機：** 客人第一次按下「確認預約」按鈕，且該 User 沒有完整資料

**流程：**

```
[客人點「確認預約」]
  ↓
[系統檢查：是否首次預約？]
  ↓ (是)
[彈出 Bottom Sheet：請留下你的資料]
  - 姓名 (必填)
  - 手機號碼 (必填，格式驗證 09xxxxxxxx)
  - 性別 (必填，男/女)
  - [確認並完成預約] 按鈕
  ↓
[資料存入 User → 預約建立 → inline 成功動畫]
```

**資料收集欄位：**

| 欄位 | 必填 | 驗證 | 用途 |
|------|------|------|------|
| 姓名 | ✅ | 2-20 字 | 老闆辨識客人 |
| 手機號碼 | ✅ | 台灣格式 09xxxxxxxx | 老闆緊急聯絡用 |
| 性別 | ✅ | 男 / 女 | 影響服務價格計算（男士剪髮 vs 女士剪髮） |

**設計決策：V2 不做 SMS OTP 驗證**

- 降低實作複雜度與新客摩擦
- 省下 SMS 服務費用與第三方整合時間
- 風險：有人可能填假電話
- 緩解：老闆可透過 LINE 訊息聯絡（不單靠電話）
- 若未來發現假電話或惡意預約成為問題，再加 SMS OTP

**二次預約以後：**
- 系統自動帶入姓名、手機、性別
- 不再跳 Bottom Sheet，直接確認預約即可

#### F-C01b: LIFF 載入階段 🆕

**問題：** LIFF SDK 初始化需要 2-5 秒（fetch profile、auth 等），V1 只顯示小 spinner，客人會懷疑「壞掉了嗎？」

**V2 解法：**

- 立即顯示店家 Logo + 品牌色背景
- Skeleton screen 模擬即將載入的預約頁結構（灰色方塊佔位）
- 底部顯示「正在為你準備預約資訊...」
- LIFF ready 後立即切換為真實內容（無白屏）

#### F-C01c: 預約成功狀態（inline） + 加入行事曆 🔧

**觸發時機：** 預約成功建立後，不跳頁，在原預約頁 inline 顯示成功狀態。

**畫面內容：**
- ✅ 成功圖示（checkmark 或植物系極簡圖示）
- 標題：「預約成功」
- 副標：「確認訊息已發送到你的 LINE」
- 預約摘要卡片：服務、日期、時間區間、金額
- 三個動作按鈕：
  - **加入行事曆** (Google Calendar deep link)
  - 查看我的預約（導向 F-C02）
  - 關閉（`liff.closeWindow()` 回 LINE 聊天室）

**「加入行事曆」邏輯（已於 V1 實作）：**

參考：[src/components/liff/booking/success-step.tsx](../src/components/liff/booking/success-step.tsx) 第 6-24 行

行事曆 event 的**時長必須對應實際服務時長**：

```typescript
// 結束時間 = 開始時間 + service.slotsNeeded 小時
const endHour = parseInt(time.split(":")[0]) + service.slotsNeeded;
const endTime = `${endHour.toString().padStart(2, "0")}:00`;
```

**範例：**
| 服務 | slotsNeeded | 開始時間 | Google Calendar event 時長 |
|------|------------|---------|--------------------------|
| 剪髮 | 1 | 11:00 | 11:00 - 12:00（1 小時）|
| 染髮 | 3 | 11:00 | 11:00 - 14:00（3 小時）|
| 燙髮 | 4 | 11:00 | 11:00 - 15:00（4 小時）|

**Google Calendar URL 參數：**
- `action=TEMPLATE` — 建立事件
- `text` — 事件標題：「{服務名} — 1008 Hair Studio」
- `dates` — 開始/結束時間（ISO 8601 格式，含 Asia/Taipei 時區）
- `location` — 店家地址
- `details` — 事件描述
- `ctz=Asia/Taipei` — 時區

點擊後會在 LINE WebView 外另開瀏覽器，跳到 Google Calendar 建立事件頁面，客人確認後事件就會加入他的 Google 日曆。

### 5.2 取消政策（基於老闆訪談確認）

```
                 預約日期是明天或之後？
                      ├── YES → 免費取消，無違規
                      └── NO  → 是當天
                              ├── 營業時間內 → 必須打電話
                              └── 非營業時間 → 線上取消但記違規
```

**老闆原話：** 「預約都可以線上，但取消你要打來。打電話有人情壓力。」

### 5.3 違規制度

| 項目 | 規則 |
|------|------|
| 違規觸發 | 當天非營業時間取消 / 爽約（NO_SHOW）|
| 違規上限 | 3 次 |
| 限制內容 | 超過 3 次 → 限電話預約 1 個月 |
| 重置 | 限制到期後自動重置 |

### 5.4 改期政策（V2 新增）

| 項目 | 規則 |
|------|------|
| 改期次數 | 每筆預約最多 2 次 |
| 改期時機 | 遵守取消政策（前天免費 / 當天營業時間打電話 / 當天非營業時間記違規）|
| 新時段驗證 | Redis lock + 可用性檢查 |
| Edge cases | 舊日期已過不可改期、並發改期+取消 race condition 由 Redis lock 保護 |

### 5.5 服務時段計算

| 服務類型 | 時段數 | 範例 |
|----------|--------|------|
| 剪髮 | 1 slot | 14:00-15:00 |

- 多時段服務必須佔用連續時段
- 系統自動檢查連續空檔

**應用層規則：** 客人首次預約時，若 `phone` 或 `gender` 為 null，則跳出 Bottom Sheet 強制收集。收集後寫入 User，之後預約自動帶入。

#### 新 Model: `Waitlist`

```prisma
model Waitlist {
  id          String   @id @default(cuid())
  tenantId    String
  userId      String
  serviceId   String
  date        String   // YYYY-MM-DD
  startTime   String   // HH:00
  status      WaitlistStatus @default(WAITING)
  notifiedAt  DateTime?
  createdAt   DateTime @default(now())

  tenant      Tenant  @relation(fields: [tenantId], references: [id])
  user        User    @relation(fields: [userId], references: [id])
  service     Service @relation(fields: [serviceId], references: [id])

  @@index([tenantId, date, startTime, status])
}

enum WaitlistStatus {
  WAITING      // 等候中
  NOTIFIED     // 已通知
  EXPIRED      // 過期（通知後客人沒在時限內回應）
  CONVERTED    // 已成功預約
  CANCELLED    // 客人主動取消候補
}
```

### 9.2 V2 新增技術考量

#### 圖片上傳（服務管理）

- Supabase Storage（V1 已有，擴展使用）
- 前端壓縮（< 5MB）
- 支援 iOS / Android 相簿選取

#### PWA 支援（加分項）

- `next-pwa` 或手寫 `manifest.json` + service worker
- 老闆可「加到主畫面」
- 實作成本約 30 分鐘
- 優先級低，Phase 2 尾聲再做

#### Waitlist 通知機制

- 當預約被取消 → 觸發 waitlist 檢查
- 查詢該時段的 waiting status 記錄（按 createdAt asc）
- 取第一個 → 發送 LINE 通知 → 更新 status 為 NOTIFIED
- 若 30 分鐘內客人未預約 → 狀態改為 EXPIRED，通知下一個

#### Mobile-First Admin

- 所有 admin 頁面預設 viewport 為手機（375px）
- 使用 Tailwind responsive classes（`md:` `lg:` 是桌面增強，不是相反）
- 測試時先在手機 viewport 測，通過後才測桌面

### 9.3 Cron Jobs

**保留 V1：**
- `/api/cron/reminders` — 發送提醒通知
- `/api/cron/cleanup` — 過期清理
- `/api/cron/at-risk` — CRM 重新分群
- `/api/cron/weekly-report` — 週報推送

**V2 新增：**
- `/api/cron/waitlist-expire` — 檢查已通知但未回應的 waitlist（30 分鐘後過期）

> ⚠️ **Vercel Hobby Plan 限制：** 每天最多 1 次 cron 執行。若需要更頻繁，需升級 Pro。V2 暫不處理，沿用 V1 策略。

## 12. 限制與已知取捨

### 12.1 V2 設計取捨

| 項目 | 決策 | 原因 |
|------|------|------|
| 不做線上金流 | 只支援現金 + 轉帳 | 老闆訪談確認現況夠用 |
| **不做 SMS OTP 驗證** | **只收手機不驗證** | **降低新客摩擦，省服務費。若濫用再加** |
| 1 小時時段 | 不改 30 分鐘 | 理髮店適用，且降低實作複雜度 |
| 只走 LINE 通知 | 無 SMS / Email | LINE 在台灣夠用 |
| 單設計師 | 不做多人排班 | 1008 是 1 人店 |
| 體驗 Onboarding | 非設定 Onboarding | 1008 客製化模式，設定由開發者完成 |
| PWA | 加分項，非必要 | 成本低但不是核心需求 |
| **移除服務後感謝訊息** | **V1 已實作但 V2 移除** | **「認真對待每一句話」，減少無意義推播** |
| Rich Menu 4 格 | V1 規劃 6 格 → V2 縮減為 2x2 | 刪除「作品集」「聯絡店家」，按鈕更大更突出 |

### 12.2 已知技術限制

- LINE Bot SDK v10 legacy Client API
- LIFF 必須在 LINE App 內開啟
- Vercel Hobby Plan cron 限制（每日 1 次）
- Upstash Redis Free Tier 限制

### 12.3 不在 V2 範圍

- 完整行銷套件（分群推播、優惠券）→ V2.2
- 評價系統 → V3
- 多設計師排班 → V2 SaaS
- 跨行業特化（醫美、寵物等）→ V4 SaaS
- 完整設計系統（只做基本版）→ 第二家店時正式化
