## 🎨 UX 取捨（設計決策需要老闆裁決）

### Q4. 拖曳建立預約時，服務下拉是否「自動依時長篩選」？ `P0`

**背景**：老闆端日曆拖曳建立預約時，可以拖出時長（例如拖 1 小時 / 3 小時）。

**方案 A：自動篩選**
- 拖 1 小時 → 只顯示：男士剪髮 / 女士剪髮 / 護髮
- 拖 3 小時 → 只顯示：染髮 / 燙髮
- 拖 2 小時 → 顯示「無符合服務」，提示調整時長
- ✅ 選錯機率接近 0
- ❌ 無法做「特殊情況：染髮壓縮成 2 小時」這類破例

**方案 B：自由選服務 + 提示**
- 下拉顯示全部服務
- 選染髮（3h）但拖的是 1h → 跳提示「這個服務需要 3 小時，要自動延長嗎？」
- ✅ 保留彈性
- ❌ 多一步確認

**要問老闆**：實際營運中會不會有「為熟客壓縮或延長服務時間」的情境？

## 💰 定價與商業模式 `P0`

### M1. 訪談已知定價錨點
- 老闆自述可接受：NT$3,000–10,000/月訂閱
- 老闆認知：市面方案「好幾萬」→ 覺得太貴、只適合連鎖
- **gap 定位**：服務單店、價位合理

### M2. 要追問的定價問題
- 試用一個月 → 正式訂閱之間，有沒有「優惠綁年約」的空間？
- 價格是含 LINE 官方帳號費用（每月推播額度）還是另收？
- 開店前置費用（設定、客戶資料匯入）要不要收一次性 setup fee？

### M3. 願付價值的真實測試
- 「如果漏接 1 個染髮客 = 損失 NT$2,000+，系統幫你一個月少漏 3 個 → 月費 NT$X 你能接受？」
- 用具體金額反算而非模糊問「願意付多少」。

### M4. 客戶的價值感受
- 老闆自己認為有用 ≠ 客戶滿意。訪談：「客戶如果還是要回到 LINE 訊息就失敗」
- 要不要在上線前先訪談 3–5 位老顧客？他們願意改用新系統嗎？

### 🎯 Q1. Demand Reality（需求真實性）`P0`

**已知證據**（訪談原文）：
- 老闆說「至少要 3,000 塊基本費用」「1 萬以下是預算」
- 市面有方案但「好幾萬塊 → 太貴 → 適合連鎖不適合我」
- 老闆自陳痛點：「基本上每天都可以排滿，**只是我花太多時間在弄**」

**YC 透鏡：這是「有興趣」還是「真需求」？**
- ⚠️ 3,000 是自報數字，不是行為數據。**沒付錢之前都不算需求**。
- ⚠️ 痛點框架是「省時間」而不是「增收入」—這類痛點通常 willingness-to-pay 比較低（因為時間省下來沒有直接兌現成錢）

**要追問老闆 `P0`：**
1. **WTP 錨定測試**：「如果每月幫你省下 10 小時回訊息時間，你覺得這 10 小時值多少錢？」（把時間量化成錢）
2. **痛點排序**：「如果只能解決一件事：(A) 省你回訊息時間、(B) 睡覺時也能收預約、(C) 客戶用起來更滿意 → 你會排哪一個第一？」
3. **反向測試**：「如果系統明天突然壞掉一個禮拜，你會回去用 Excel，還是會打電話來煩我？」（前者＝nice-to-have，後者＝真需求）
4. **既有付費意願證明**：「你過去 3 年有沒有為這個問題真的付過錢（例如買過任何管理 app、請過工讀生）？」

### 🔍 Q2. Status Quo（現狀成本量化）`P0`

**已知證據**：
- 手機 LINE + Google/Excel 交叉比對
- 燙/染客人臨時取消「真的會比較麻煩，卡幾個小時」
- 每天排滿六七成，挽髮時客人覺得「好專業」但老闆覺得可以更好

**YC 透鏡：現狀的真實成本還沒量化**
- 「花太多時間」= 每天幾分鐘？每週幾小時？**沒有具體數字，無法反推 ROI**。
- 漏接的燙染客一次＝多少錢？一個月發生幾次？

**要追問老闆 `P0`：**
5. **時間量化**：「明天一天，你回預約訊息＋抄 Excel 總共花了多少分鐘？」（請老闆實測一天，不要用記憶估算）
6. **燙染機會成本**：「過去 3 個月有幾次臨時取消讓燙染時段完全空掉？一次損失約多少錢？」
7. **失眠成本**：「睡覺時漏掉的預約訊息—早上起來有沒有因為『客人已經約別家』而失單的案例？多久一次？」
8. **Excel 具體給我看**：「可以把過去 1 個月的 Excel 給我截圖嗎？我想看實際混亂程度」（這會暴露隱藏痛點 e.g. 重複預約、記錯時段）

### 👤 Q3. Desperate Specificity（誰最痛）`P0`

**已知證據**：
- 老闆自己：單人理髮廳、有小孩、一週三天沒課想陪小孩、有休假
- 其他同業：美髮 / 美甲 / 美容 / 中醫（訪談順便提到）

**YC 透鏡：N=1 不夠**
- 老闆自己是 single data point。他覺得 3-10k OK，不代表其他同業也一樣。
- 「類似業態都能用」是創辦人的假設，不是驗證。

**要追問老闆 `P0`：**
9. **介紹同業**：「你願不願意幫我介紹 2 個同行（美髮 / 美甲 / 個人工作室）讓我也訪談？」（如果他不願意 → 痛點其實沒那麼強）
10. **他心中的理想客群像**：「你覺得這個產品最適合什麼樣的店？一人店？兩人店？有請助理的？」（幫你切 ICP）
11. **客戶端的聲音**：「你願不願意幫我問 3 位最熟的老客戶，對現在 LINE 訊息約時段的感受？」（產品成敗最終在客戶，不在老闆）

### 🎯 Q4. Narrowest Wedge（最窄切入點）`P1 → 部分 P0`

**已知證據**：目前 V1 功能已經很完整（LIFF 預約、取消改期、違規機制、CRM 分群、行銷推播、週報、日結算...）

**YC 透鏡：⚠️ 你已經 over-built 了**
- 如果最窄 wedge 是「睡覺時也能預約 + 自動回覆」，其實 MVP 只需要：LIFF 預約頁 + 時段鎖 + LINE 通知。
- CRM 分群、違規機制、行銷推播、週報 → 這些都是 wedge 擴展後才有價值的功能。
- **老闆訂閱會不會因為「功能太多用不到」而覺得貴？** 值得測試。

**要追問老闆 `P0`：**
12. **功能優先級排序**：把目前 V1.1 所有功能列出來讓老闆「刪到只剩他真的會每天用的 3 個」—這 3 個才是真 wedge。
13. **功能陷阱測試**：「如果只有『LINE 預約 + 後台看時段表 + LINE 推播提醒』，沒有違規機制、沒有 CRM、沒有週報—你月費願意付多少？」（把功能拆開問定價）

### 👁️ Q5. Observation & Surprise（實地觀察）`P0`

**已知證據**：
- 老闆同意「測試兩三週 → 試用一個月 → 雙方滿意再訂閱」
- 但目前還沒真的讓老闆與其客戶跑過一次完整流程

**YC 透鏡：⚠️ 這是整個專案最大的風險**
- 訪談本身是「老闆腦中的想像」，不是「老闆實際用過」。
- 客戶端更是完全沒驗證 — 老顧客是否願意放棄 LINE 直接訊息改點 LIFF？**這會是最大失敗點**。

**要追問老闆 `P0`：**
14. **Beta 排程**：「何時可以開始讓你的真實客戶用？先拿 5 個老顧客試一週，我坐旁邊看你跟客戶的反應」
15. **「我坐旁邊看」請求**：訪談提到「有一種心理因素是我們考量不到的」—**必須親自到店觀察**老闆操作後台、客戶用 LIFF 的反應，不能只看數據。
16. **失敗情境**：「如果客戶試用後還是回頭傳 LINE 訊息給你約時段，你會怎麼處理？」（測試老闆是否願意「強制客戶改習慣」，不願意的話產品會失敗）

### 🔮 Q6. Future-Fit（未來適配）`P1`
