# V3.8 — Admin App Consolidation Plan

> **Status**: Phase 1 Audit 完成，等老闆 review 後才執行 Phase 2
> **Author**: 2026-04-30
> **Trigger**: 多年累積後 admin 後台頁面/API 出現重疊，需整併
> **Scope**: admin 後台頁面 + admin API endpoint 整併
> **Out of scope**: 行銷整合（顧客分群 + 推播 + 券）— 老闆已要求暫緩；客戶資料清理（v3.8 customer-data-hygiene 另案）；V3.6 報表（剛上線，保留）

> **重要**：本 plan 跟 [docs/v3.8-customer-data-hygiene-plan.md](v3.8-customer-data-hygiene-plan.md) 並存。兩者命名都用 v3.8 但 scope 不同：
> - **本 plan（app-consolidation）**：精簡頁面/API，砍重複
> - **customer-data-hygiene**：ghost user 清理 + customer 搜尋修

---

## 1. 為什麼要做這個 plan

### 1.1 觸發點

老闆透過截圖反饋 admin 介面太多功能、太多重疊頁面。盤點後發現：

- **22 個 admin route + 26 個 admin API endpoint**
- 至少 **3 對頁面功能高度重疊**（/dashboard ↔ /reports daily / /analytics ↔ /reports monthly / /payments ↔ /cash-flow）
- 至少 **2 個死 endpoint 候選**（前端 0 引用）
- 至少 **1 個孤兒元件**（retention-push-widget）
- **/analytics 已被 V3.6 報表完全取代**（page title 自己都寫「報表」）
- 綠界 ATM + Tier S 預付 — 老闆確認**沒在用且不會用**，UI 還掛著

### 1.2 整併目標

1. **降低老闆認知負荷** — sidebar 從 13 項降到 6-8 項
2. **消除「同一件事兩個地方做」的混亂** — 例如「待對帳」目前出現在 3 頁
3. **砍掉 dead code** — 死 endpoint、未使用元件、deprecated 欄位 UI
4. **保留有 value 的雙視角** — 例如 /payments 末五碼搜尋是專家工具，跟 /reports daily 互補

### 1.3 非目標

- 不重做 V3.6 三視角報表（剛上線，觀察期）
- 不整合行銷工具（老闆暫緩）
- 不動 booking / payment / settle / LIFF 核心流程
- 不動 cross-tenant 安全邏輯
