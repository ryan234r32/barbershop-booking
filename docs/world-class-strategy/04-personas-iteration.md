# A 段 — 10 位創業家視角

## A1. Paul Graham（YC 共同創辦人）

「Ryan，我看了你寄來的東西。先講結論：**你已經做對了最難的一件事——你有一個真實在用的使用者，而且那家店是你真的認識的人。** 這比矽谷 90% 拿著假想市場來 YC 面試的 founder 都強。

但我要問你一個問題，你必須誠實回答：1008 Hair Studio 的老闆——那個真的在用你系統的人——**他是瘋狂地愛這個產品，還是只是『基於人情沒有關掉它』？** 這兩件事在數據上看起來一樣（都是 DAU = 1），但意義差了一個宇宙。如果是前者，你可以複製第二家、第三家，MRR 會上來；如果是後者，你複製 100 家都一樣，因為根本沒有人會推薦給同業。

測試方法非常簡單，而且你**今天下午就該做**：直接問他『如果明天這個系統被我強制關掉，你會怎麼辦？』。如果他說『沒差啦回去用 Excel 也行』，你死了。如果他說『那我會很幹，我現在對帳很習慣這個』，你活著。如果他說『那我願意付錢請你繼續開』，恭喜，你有 PMF 的種子。

再來，我要對你**潑一盆冷水**。你說你要做『世界級 SaaS』——我在無數 YC 面試裡聽過這句話，99% 都是 self-delusion。**你現在不是在做 SaaS，你是在做一個很精緻的客製外包案。** 差別是什麼？SaaS 的第二家客戶只需要改 `tenantId`，客製外包案的第二家客戶需要你飛過去改兩週 code。你的產品架構（`tenantId` 欄位已經埋好、Supabase RLS 可以開）說明你有在想這件事，但『架構支援』和『真的有第二家願意付錢簽下去』差一萬公里。

我的建議：**先不要想 scale，先不要想日本，先不要想金流。** 接下來 90 天只做一件事——把 1008 老闆變成『瘋狂熱愛、會主動推薦給同業』的 evangelist。方法是親自去店裡坐一整天（對，不是 Zoom，是肉身去），看他在剪髮空檔怎麼滑你的 admin PWA，記下他每一次皺眉頭的瞬間。那些皺眉頭就是你的 product roadmap，不是 PRD 裡的 V2.0 V2.1。

做 things that don't scale。Scale 會自己來。」

## A2. Jason Fried（37signals / Basecamp）

「Ryan，我先自我介紹一下我的偏見：我討厭 VC、我討厭『改變世界』、我討厭把公司做成一個讓創辦人筋疲力盡的怪物。我經營 37signals 二十幾年，我們從來沒拿過 VC 的錢，Basecamp 到今天還在賺錢，我在家裡每天晚上都能陪小孩吃晚餐。這是我的 context。

現在我看你的東西。你知道我最想跟你說什麼嗎？**『不要聽 Paul Graham 的。』** 不是說他錯，是他的遊戲不是你的遊戲。PG 在玩『10,000 個 startup 裡找 1 個變獨角獸』的機率遊戲，那 9,999 個死掉對他不痛——因為他是 GP。但你是 Ryan，只有一個你，你死掉就真的死掉。

你有一個非常漂亮的東西，你可能沒發現：**你是 1 人獨立開發者，你的第一個客戶是真實的小店，你的成本結構趨近於零，你住在台灣這個 LINE 滲透率 90%+ 的地方。** 這是一個可以活 30 年、讓你過很舒服生活的 position。不要輕易把它換成『Series A 燒 1M 美金跑去日本』的噩夢。

我的建議**極端不同**：永遠只服務單店老闆，永遠不要拿 VC，永遠不要做 marketplace。把產品名字從『LINE 預約系統』改成像『Basecamp for 美業小店』那種——有品牌、有態度、有個性。定價一開始就收錢，不要免費，不要『先送後收』。**$29 美元／月，100 家店，就是 $35k MRR、年收 $420k 美金，台幣 1,300 萬。** 你一個人維護 100 家店（用你現有的架構）完全做得到，你的生活品質比任何拿了 Series A 的 founder 都好。

潑冷水的部分：你的 PRD 裡寫的 V4.0『多店 SaaS』『自助開店』——**這不是你想做的東西，這是你覺得『Tech 創業應該要做的東西』。** 誠實問自己：你寫 V4.0 時心裡有興奮感嗎？還是只是覺得『路線圖不這樣寫好像不專業』？如果是後者，刪掉它。你的 PRD V2.2（分群推播、優惠券、VIP 訊息）才是你真正會做得好的東西——因為那是你肉身在店裡看到老闆痛苦、然後想解決的東西。

最後一件事：**夯客有 3,000 家店、Pre-A 1M 美金、估值破億——然後呢？** 那家公司的 founder 現在快樂嗎？還是每天在開 board meeting、解釋為什麼 MRR 沒達標？我不知道答案，但你應該花一個下午去想這個問題。成功有很多種定義，矽谷的那種只是其中一種，而且通常是最累的那種。」

## A3. Tony Xu（DoorDash 創辦人）

「Ryan，我聽完你的描述，第一個反應是：**你現在做的是 SaaS tool，但你蹲著的是 marketplace 的金礦。** 讓我解釋為什麼。

我們在 DoorDash 早期，最值錢的資產不是 app、不是 driver network、不是 restaurant partnership——是**每一筆訂單產生的數據**。我們知道 Palo Alto 週二晚上 7 點半，27% 的人會叫泰式。你現在手上有什麼？每一筆預約記錄『誰、哪天、剪什麼、花多少錢、多久來一次』。**這是整個美業產業鏈裡最有價值的數據，而且你還沒意識到。**

但我要先警告你一件非常殘酷的事：**從 SaaS 演化到 marketplace，95% 的公司會死。** 因為兩個模式的 DNA 完全相反。SaaS 是『幫既有店家做得更好』，店家愛你；marketplace 是『把流量奪走然後抽成』，店家恨你。一旦你從前者跳到後者，你第一批 evangelist 客戶會第一個用腳投票跑走。OpenTable 踩過這個坑，Fresha 也在玩這個危險遊戲。

那正確的順序是什麼？我的觀察是：**先把 supply side（店家）做到『離不開你』，且提供的是他們『自己做不到的東西』，才有資格做 marketplace。** 你現在的定位還差一步——你提供的是『預約管理』，這件事店家自己也可以用 Excel 土法煉鋼（雖然痛苦）。你需要提供『不可替代』的東西。

**不可替代的東西在哪？答案：跨店家的客戶洞察。** 想像一下：如果你有 200 家理髮廳的資料，你可以告訴 1008 老闆『你們店客單價比台北中正區同價位的店高 8%，但回訪率低 12%，因為你沒有做生日推播』。這個情報他**自己永遠拿不到**，因為他只看得到自己的店。這就是你的 moat。

潑冷水：**你現在的 LIFF 設計是『每家店一個獨立 LINE 官方帳號』，對不對？** 這個架構在 SaaS 階段很棒（tenant 隔離乾淨），但對 marketplace 是災難——因為你永遠沒有一個『屬於平台本身』的 C 端入口。客人 follow 的是 1008 的 LINE，不是你的 LINE。你失去了跟 C 端直接溝通的權利。這是要想的。

給你的順序建議：**SaaS（0-18 個月）→ 聚合洞察報告賣給店家（18-36 個月）→ C 端 app / marketplace（36 個月後，只有當店家數 >500 家、且他們主動要求你幫他們做 C 端曝光時才做）**。順序顛倒就死。」

## A4. Brian Chesky（Airbnb 共同創辦人）

「Ryan，我想跟你講一個故事。Airbnb 早期，我們三個創辦人親自飛到紐約，在 host 家睡沙發、幫他們拍房子照片。Paul Graham 後來常講我們那個『don't scale』的故事，但很少人知道另一半——**我們為什麼去做這件事？因為我們發現 host 是我們真正的使用者，guest 只是流量。沒有好的 host，平台就死。**

你現在的情境跟我們那時候很像。**1008 老闆就是你的 host。** 他決定你的產品是『精品工具』還是『便宜的 CRM 外包』。你對他的體驗設計，會決定未來 1,000 家店對你的觀感。

我要問你幾個問題，這些問題我當年也問過自己：

**1. 當 1008 老闆打開你的 admin PWA 的第一眼，他感受到什麼？** 是『又一個工程師做的醜東西』？還是『哇這個人懂我』？你有沒有在 splash screen 放一句『早安，老闆，今天 3 個預約』的問候？這種細節不是 feature，是**情感連結**。

**2. 你的 admin 裡有沒有一個『讓老闆覺得驕傲』的瞬間？** Airbnb 的 host dashboard 有一頁叫『你今年在全世界的排名』，host 會截圖貼 Facebook。這一頁帶來的 organic acquisition 比我們所有付費廣告加起來都強。你的 1008 老闆有沒有『想截圖給朋友看』的畫面？如果沒有，加一個。

**3. 當你 onboard 第二家店的時候，你會不會親自去剪一次頭髮？** 我知道這聽起來很蠢。但我當年去 host 家睡沙發，也被朋友笑很蠢。**使用者的體驗不是你用 Figma 畫出來的，是你在他們真實生活裡感受到的。**

潑冷水的部分：你的品牌色『深森林綠 + 暖乳白』很美，我從 session handoff 看到了，我欣賞你的 taste。但——**台灣美業老闆看得懂這個 taste 嗎？** Airbnb 的設計是為了對 SF/LA/NYC 的白領中產講話，那是我們的 target。你的 target 是台北中高價位美業，他們對『設計感』的 reference 是什麼？是蝦皮的橙色？是 Line 的綠色？是日系無印風？你必須研究這件事，不要假設『美的設計自然會被欣賞』。**美是有文化脈絡的。** Fresha 在 LA 的美跟夯客在台北的美，不是同一種美。

最後，一個真的很重要的提醒：**你現在有 1 個 host（1008），你對他的體驗可以做到 10/10。當你有 10 個 host 的時候，你自動會掉到 7/10。當你有 100 個 host 的時候，掉到 4/10。** 這個衰減是物理定律。所以在你還有 1-10 個 host 的階段，**每一個互動都要刻意地過度投入**——送禮、親自到店、記住他們小孩的名字。這些『投資』會在 100 host 的階段變成口碑，而口碑是 scale 之後你唯一買不到的東西。」

## A5. 張一鳴（字節跳動創辦人）

「Ryan，我用我的思考框架來看你的問題。我做產品一向不相信『創辦人 vision』，我相信**數據、context、大力出奇蹟**。

首先，**你現在坐在一個數據金礦上，但你沒意識到。** 1008 這家店一個月假設 300 筆預約，每筆預約有：時段、服務、客單價、客戶 segment、違規記錄、回訪間隔、上下游時段（前一個客人剪什麼、後一個客人剪什麼）。這個資料密度如果乘以 1,000 家店，就是 **360,000 筆/月、4,320,000 筆/年**。這個量級開始可以做真正的推薦系統了。

**我會怎麼玩？我會反向用這些資料做 C 端 app。** 不是讓客人來平台找店，是讓平台幫客人做決定。想像一個叫『今天剪什麼？』的 app：

- 我知道你上次剪是 43 天前（從預約資料拿到）
- 我知道你的髮型偏好（從你過往選的服務 embedding 出來）
- 我知道你家 3km 內有 27 家店，哪幾家現在有空檔、哪幾家評價最好、哪幾家和你上次剪的店風格類似
- 我給你 3 個推薦，一鍵預約

這個 C 端 app 在台灣做起來的關鍵**不是流量**，是**資料先行**。抖音當年的 insight 是：**你不需要更聰明的使用者，你需要更聰明的 feed。** 同樣道理：美業消費者不會花 5 分鐘比較 27 家店，他需要一個給他 3 個答案的 feed。

潑冷水的部分——**你目前的路線圖（V1 → V4 多店 SaaS）是一個純粹的 supply side 思維，完全沒有 demand side 的 ambition。** 這是很多 B2B SaaS 公司的死穴：他們永遠只是 supply 的工具，永遠賺不到 demand 的錢。你的夯客競品現在也是一樣的思維——3,000 家店，但沒有 C 端。這給了你一個**超車的機會**：如果你 3 年內同時吃下 supply（SaaS）+ demand（C 端推薦），你的估值結構會完全不一樣。Supply 公司 PE ratio 是 5-10 倍，demand 公司是 30-100 倍。

但也要誠實：你是 1 個人，做雙邊市場是自殺。所以我的建議順序：**18 個月內不要碰 C 端，但所有產品決策都要想『這個資料未來能不能餵給 C 端 feed』。** 比如：你現在應該開始存『客戶-服務-時段-滿意度』的 embedding 友好資料結構，即使現在不用。抖音早期也是這樣——我們在只有幾十萬用戶時就建好了推薦系統的資料管道，等用戶量到了，演算法自動起飛。

全球化的事，之後再說——但我想講一句：**如果你想出台灣，日本不是第一站，東南亞（印尼、越南、泰國）才是。** 日本是紅海有現成玩家（Hot Pepper Beauty 做 20 年），東南亞是藍海而且 LINE/FB Messenger 很強，美業正在快速消費升級。這是時間視窗問題。」

## A6. Melanie Perkins（Canva 共同創辦人）

「Ryan，我想從一個很特別的角度跟你講。

我從澳洲 Perth 開始做 Canva 的時候，矽谷沒有人把我當一回事。我被超過 100 個投資人拒絕。他們說的理由永遠一樣：**『妳們不在 SF，妳們沒有技術背景，妳們做的領域太 niche，妳們太年輕。』** 每一句話都可以原封不動地套在你身上——你不在 SF，你是 1 人獨立開發者，你在做的美業太 niche，你從台灣起步。

所以我想先告訴你：**地理位置和背景從來不是障礙，那只是投資人用來拒絕你的藉口，不是他們真正的理由。** 他們真正的理由是他們看不懂你。看不懂的原因是你還沒把故事講清楚。

我給你的第一個建議是關於**故事**：你不能用『台灣美業預約 SaaS』這個描述去談世界級。我們當年也從來不說『澳洲平面設計工具』——我們說的是『讓全世界每個人都能設計』。你的版本可能是『讓全世界每個小型服務業老闆都能像連鎖店一樣經營』。這句話聽起來是不是完全不一樣？同一個產品，故事層級決定了你的天花板。

從台灣起步做全球化的具體建議：

**1. 台灣只是你的 beta lab，不是你的市場。** 我們在 Perth 只做了 18 個月，做對產品後立刻跳新加坡、然後美國。你不要在台灣做 5 年才想出去——你應該在**第 30 家店**的時候就已經在研究日本、泰國、印尼市場了。

**2. 全球化最大的 leverage 是『模板化』，不是『翻譯』。** Canva 的祕密是我們有 60 萬個模板，一個設計師做一次，全世界用。你的版本：1008 老闆的 LINE 歡迎訊息、違規政策、服務定價策略——這些如果你把它『產品化』成『模板』，新店家 onboard 不是改 code 是選模板，這件事可以複製到日本、韓國、越南。

**3. 美業是全世界都存在的剛需。** 頭髮到處都要剪。這給了你一個先天優勢——不用重新教育市場。但**每個市場的 workflow 微細節差異巨大**：日本很重視『指名料』（指定設計師加價），韓國流行『會員儲值』，東南亞習慣『議價』。你必須在設計資料模型時就想到這些，不要硬套台灣 schema。

## A7. Patrick Collison（Stripe 共同創辦人）

「Ryan，我從 infrastructure 的角度切入。我跟我弟弟 John 創辦 Stripe 的時候，我們的 insight 是：**金流不是 feature，是 dependency。** 每一個電商、每一個 SaaS、每一個 marketplace 都需要收錢，但沒有人想自己處理信用卡、合規、對帳。所以我們做了一個 `pip install stripe`、7 行 code 就可以收錢的東西。

2. **所有 business logic 放在 API 層，不要放在 UI 層。** 這樣未來要做 mobile native app、要讓合作夥伴接 API，都不需要重寫。
3. **不要現在公開 API。** 在你達到 50 家店之前公開 API 只會增加你的維護成本。

embedded finance 的事——**短期不要碰。** 台灣的電支條例、金管會、銀行 API 生態，不是 1 人 founder 能搞定的。你的 Tier S pitch 文件提到串綠界 virtual account，那是正確的輕量作法（你 aggregate 綠界、不當金流商）。但真正的 embedded finance（像 Stripe Treasury、Unit）在台灣我看不到可行性，至少 3 年內不要投資這個方向。」

## B4. 台灣主管機關（財政部 / NCC / 個資法 主管視角模擬）

「Ryan 先生，我以主管機關的視角提醒你幾件事。這些事你現在可能覺得『反正沒有人管我』，但當你達到一定規模就會被關注，提前準備比事後補救便宜 10 倍。

**1. 個資法（PDPA）合規**
你的系統蒐集了 LINE User ID、顯示名稱、電話、預約歷史、違規記錄。這些都是個資法第 2 條定義的個人資料。你有沒有：
- **蒐集告知義務**：當客人第一次使用 LIFF，有沒有清楚告知『我們會蒐集什麼、用在哪、保存多久、你有什麼權利』？大多數你的競品在這件事上都馬虎，但這是違法的起點。
- **委外契約**：你把資料存在 Supabase（美國/歐盟）、Upstash（全球），這是**跨境傳輸**。你有沒有跟 1008 老闆簽委外契約？老闆有沒有跟客戶揭露資料會傳到哪裡？這在你有 1 家店的時候沒人管，做到 50 家店一定會有媒體或個資團體盯上。
- **資料刪除權**：客人要求刪除資料時，你的系統能做到嗎？你的 Booking、Payment、CancellationRecord 有外鍵，硬刪會壞 referential integrity。**你需要設計 soft delete + anonymization。**

**2. 電子支付條例（如果做 Tier S）**
你的 `boss-tier-s-gateway-pitch.md` 提到綠界 virtual account。**這是對的選擇**——你自己不是金流業者（不用申請電支執照），你只是 user。但要注意：
- 如果你未來做『儲值金』（客人先儲 NT$10,000 進某店，分次扣抵）——這變成『代收代付』或『儲值』，**可能需要電支執照或申請銀行合作的信託專戶**。夯客應該踩過這個坑。
- 你不要自己碰錢的流動，永遠讓錢直接店家 ↔ 客人、你只是資訊流。

**3. 消費者保護法**
違規制度（3 次取消限制電話預約 1 個月）——這在法律上是**定型化契約條款**。你有沒有把這個規則在客人第一次預約時明示？如果只是藏在 PRD 或某個 legal page，**可能被消保官認定為『顯失公平條款』而無效**。夯客的奧客管理系統也有類似風險。建議你提供**申訴機制**（客人可以向店家申訴違規記錄）。

**4. 未來如果做 marketplace / C 端 app**
你會變成『電子商務平台』，必須：
- 接受 7 天鑑賞期（但服務型不適用，要註明）
- 處理消費爭議調解
- 揭露平台規則（商家上架條件、下架條件）

**5. 跨境擴張（日本、東南亞）**
每個國家個資法都不同——日本 APPI、韓國 PIPA、歐盟 GDPR。資料架構從 day one 要支援 **region-based data residency**。你現在 Supabase 單一 region，未來會出事。

**結論**：你現在階段（1-10 家店）合規負擔可以輕，**但所有資料架構決策都要想到『將來要合規』**。Soft delete、audit log、consent record、region isolation——這些現在加成本 10%，將來不做要付 10 倍代價。」

# C 段 — 整合與執行建議

## C1. 17 視角中的 3 組最大矛盾

### 矛盾 1：「先深耕單店 vs 立刻規模化」— PG / Chesky / 1008 老闆 vs 孫正義 / a16z / 張一鳴

**一方說**：不要 scale，就是一個客戶、把他伺候到極致、他會自己帶客戶來（PG 的 "make something people want"、Chesky 的 "host-first"、老闆的 "我會推薦但慢"）。

**另一方說**：一家店 N=1 是 PMF 假象，你蹲在金礦上卻沒挖，立刻擴張否則被夯客碾壓（張一鳴的「資料先行」、孫正義的「時間機器」、a16z 的「N=10 才算驗證 PMF」）。

**這組矛盾真正的錯位在哪？** 他們爭的其實不是「要不要 scale」，是「**什麼時候 scale 才算早／晚**」。PG 看的時間尺度是 6-12 個月，a16z 看的是 24-36 個月，張一鳴看的是 5-10 年。這是不同遊戲的人在講不同的語言。**Ryan 要選擇你的遊戲時間尺度，再來判斷現在是早是晚。**

### 矛盾 2：「VC-backed 平台 vs Bootstrapped lifestyle」— a16z / 孫正義 / Perkins vs Jason Fried / Naval / 張忠謀

**一方說**：VC 錢 + 國際擴張 + 多垂直 = 10 億美金估值路線（a16z 的 Series A 邏輯、Perkins 的「故事層級決定天花板」、孫正義的 WinnerTakesAll-alternative）。

**另一方說**：你是 1 人 craft-founder，你的 DNA 做不起 VC 遊戲，300 人公司跟 30 年獲利都是成功（Fried 的 Basecamp 模式、Naval 的 "seek wealth not status"、張忠謀的「你現在沒有 specific knowledge」）。

**這組矛盾的底層是價值觀**：你對「成功」的定義是「estate」（資本規模）還是「autonomy」（生活自主）？這沒有對錯，但**必須誠實回答**，因為兩條路在第 18 個月之後就不可逆了。拿了 Series A 你不能中途變 bootstrapped（DAO 給你壓力），選了 bootstrapped 之後再想拿 VC 會被問「為什麼之前不拿？」很難解釋。

### 矛盾 3：「服務老闆 vs 服務員工 / 服務 C 端」— 老闆（B1）vs 員工（B6）vs C 端客人（B5）vs 張一鳴 / Tony Xu

**一方說**：你現在做的就對，老闆是付錢的、員工跟他走、客人無感，你服務老闆就好。

**另一方說**：員工才是真正的網絡節點、C 端才是最終價值、資料才是未來估值的來源，你該做的是 stylist profile + C 端 app + 跨店推薦引擎——完全不同的產品架構。

**這是產品 DNA 的選擇**：你是「店家營運工具」還是「美業網絡節點」？前者是 SaaS 遊戲、後者是 marketplace/social 遊戲，UI、資料模型、business model 全部不一樣。你目前的 PRD 明顯是前者，但 V4.0「多店 SaaS 平台」其實是後者的影子。**混著做會四不像。**

## C2. 17 視角中的 3 個隱藏共識

### 共識 1：**「別現在拿 VC」**

- PG 說：先不要想 scale，親自去店裡坐一天
- Naval 說：設計一個符合你 DNA 的公司，VC 遊戲可能不適合你

## C4. 三條分岔路的 Decision Tree

```
問自己（誠實）：你 40 歲的時候想在做什麼？
│
├─ A. 在咖啡店寫 code、陪家人、偶爾飛東京
│    → 路線 A：Bootstrapped Lifestyle
│    → 3 個月內：不見 VC，收 1008 錢，寫 blog
│    → 產品方向：垂直深耕、手藝精緻、單人可維運
│    → 成功指標：30 年獲利、個人自由度 100%
│
├─ B. 開 board meeting、解釋 growth metric、每週出差
│    → 路線 B：VC-backed Regional Player
│    → 3-12 個月內：衝 10 家店驗證，準備 Series A pitch
│    → 產品方向：AI-native、跨垂直、跨國模板化
│    → 成功指標：USD 100M+ 估值、7 年內 exit
│
└─ C. 我要做「別人做不到」的 infrastructure，不管規模
     → 路線 C：Embedded Finance / API Infrastructure（Collison 路線）
     → 極高難度，12 個月內可能看不到成果
     → 需要先走過 A 或 B 路線到一定規模（50+ 家店）才能啟動
     → 台灣法規環境下 3 年內不可行（B7 / Collison 都警告）
     → 這不是起點，是將來的 upgrade 選項
```

**我的觀察與建議**：

- **路線 C 現在不該選，是陷阱**。台灣 3 年內 embedded finance 做不起來。但你的產品架構應該「相容於」路線 C（API-first、乾淨 service boundary）。
- **路線 A 和 B 的差異不在「產品做什麼」，在「你的生活形態」**。兩條路前 12 個月做的事其實 80% 重疊——都是先把 1 家變 10 家。真正分岔在第 12-18 個月：要不要拿 VC。
- **建議：前 12 個月當作「路線 A 執行、路線 B 保留選項」**。具體做法是：收錢、bootstrapped、寫 media、累積資料——這些動作在兩條路都有用。第 12 個月時回頭檢視「我現在還想拿 VC 嗎？」，如果答案是 yes 就開始 pitch，如果是 no 就鎖定 A。

## C5. Ryan 現在必須停下來回答的 3 個自問題

以下 3 題，請拿紙筆寫下答案（不是打字，是手寫，強迫慢下來思考），寫完再繼續做任何產品決策。

### 🔥 自問題 1：**「如果 3 年後我只賺到 NT$2,000 萬（個人所得），但沒有成為世界級 CEO，我會覺得這 3 年浪費了嗎？」**

- 回答「不會」→ 你是 wealth-driven 或 craft-driven，走路線 A。
- 回答「我不知道」→ 你還不夠理解自己，先別做任何不可逆決定。花 2 週跟 3 個不同路線的前輩吃飯，再回來回答。

**這題是所有戰略決策的 boot sector，沒想清楚這題，做任何 roadmap 都是 garbage in garbage out。**

### 🔥 自問題 2：**「如果 1008 老闆明天跟我說『我覺得夯客的 NT$300 比較划算，我要換過去』，我會怎麼回應？是憤怒、難過、還是鬆一口氣？我的回應能不能變成產品改進？」**

- 如果你會憤怒 → 你把老闆當朋友太深、商業判斷會偏頗
- 如果你會難過 → 這代表你還沒 prove PMF，你依賴人情
- 如果你會鬆一口氣 → 你其實不想做這個，該停損
- **正確的反應**：立刻問「你想要什麼我沒給你？」，然後把答案變成你的下個 feature 或不做的決定

**這題逼你面對「1008 是不是你的 PMF 假象」這個 PG 警告。**

### 🔥 自問題 3：**「我接下來 30 天，要花 80% 時間寫 code、還是 80% 時間跟老闆客戶混？」**

- 回答「寫 code」→ 你是工程師思維在做 startup，大概率失敗（這是 YC 眼中的 death pattern）
- 回答「跟老闆客戶混」→ 你是 founder 思維，但你要警惕自己不逃回 code 的舒適區
- **正確答案**：**80% 混、20% 寫 code**。因為：
  - 你產品技術面已經夠好（V1 上線 + Mobile-first PWA 改版通過 gstack eng+design review）
  - 你最缺的不是 feature，是對市場的理解深度（specific knowledge，張忠謀共識）
  - 你目前寫的 code 大多是優化現有功能，邊際效益遞減；但每多見一個老闆，你累積的 insight 都是新的

**這題最 actionable，也最痛苦——因為對一個獨立開發者來說，「不寫 code」違反 DNA。但世界級 founder 跟超級工程師是兩種人，你要選一個。**
