## 0. 給新 session 的 onboarding context

### 0.1 為什麼要做 V3.6

V3.5 demo 後使用者反饋（4/27 收尾討論）：
1. Hero 估算金額 NT$26-66 萬 — 老闆會質疑「哪來的？」，過於 sales pitch

6. 缺少**月目標達成率**、**警報 banner**、**自然語言摘要**

V3.6 用使用者提供的 2 份 HTML mockup（monthly + annual）+ 設計規格書，重做整個 /reports 結構：3 個視角 tab，各有獨立 mental model。

## 1. 戰略目標

### 1.1 為老闆解決什麼

每日 → **不用打開 Excel**，今天的對帳 + 明天的預約都在儀表板看完
每月 → **看到問題就知道 action**，不用看完還在想「所以呢？」
每年 → **可以列印給合夥人看**，正式報表格式（PDF-ready）

### 1.2 真實營收影響（保守估算）

引入 V3.6 後 3 個月，預期：
- **離店再預約率**從 ~20%（猜）→ 50%+ = 多 30% 客戶 rebook = +NT$30-40 萬/年
- **月目標達成率視覺化** → 老闆會主動推到目標 = +5-10% 月營收
- **警報凌駕一切** → 早 1-2 個月發現 KPI 下滑 → 介入早 = 救回 NT$10-20 萬

V3.5+V3.6 累積估算：**+NT$60-130 萬/年**（與 V3.5 的 26-66 萬 overlapping）。

### 1.3 非目標

- 多店 SaaS（V4）
- AI 自動寫報告（V5+）
- 自定義報表（讓老闆自己拖拉指標）— 過度設計，先給固定 3 視角

## 4. Phase B — 後端新指標

### 4.1 必要的新指標（mockup 上有但程式沒做）

| 指標 | 算法 | 資料需求 |
|---|---|---|
| **離店再預約率** (Pre-book Rate) | 客戶結帳當下（status=COMPLETED）有沒有在系統內建立未來預約 | ⚠️ 需要新 instrumentation |
| **月目標達成率** | 當月實績 ÷ 月目標 | 需要 `Tenant.monthlyTargets` 欄位 |
| **YoY 12 個月對比** | 今年 + 去年同月雙 series | 需要 2024 Excel 匯入 |
| **RFM 分群** | 三維（Recency/Frequency/Monetary）矩陣 | 計算邏輯重做 |
| **Alert 規則引擎** | 5 條 rule，輸出 0-3 條警報 | 純計算 |

### 4.2 Schema 變更

```prisma
// Tenant.featureFlags 已存在，加 monthlyTargets jsonb
model Tenant {
  // ...
  monthlyTargets Json? @map("monthly_targets")
  // {"2025-12": 130000, "2026-01": 140000, ...}
}

// Booking 加 prebookSource — 標記是否「離店時建立的下次預約」
model Booking {
  // ...
  prebookSource Boolean @default(false) @map("prebook_source")
  // 如果這筆 booking 在「結帳完當天 + 0-2 小時內」由 admin 建立 → true
}
```

### 4.3 新 aggregate 函數

加到 `src/lib/reports/aggregate.ts`：

```typescript
export async function computePrebookRate(tenantId, range): Promise<number>
export async function computeAlerts(tenantId, range): Promise<Alert[]>
export async function computeRfmSegments(tenantId, range): Promise<RfmGrid>
export async function computeYoYTrend(tenantId, anchor): Promise<YoYData>
export async function computeMonthlyTarget(tenantId, range): Promise<TargetData>
export async function computeNaturalLanguageSummary(data): string  // 純規則式產文字
```

### 4.4 RFM 分類規則（要老闆確認）

| 分群 | Recency (天) | Frequency (次/年) | Monetary (NT$/年) |
|---|---|---|---|
| 🏆 **冠軍** (Champion) | <30 | ≥6 | ≥6000 |
| 💎 **忠實老客** (Loyal) | <60 | ≥4 | ≥3000 |
| 🌱 **新客觀察期** (New) | <90 | 1-3 | 任意 |
| ⚠️ **流失中** (At Risk) | 60-180 | ≥2 | 任意 |
| 🚪 **已流失** (Lost) | >180 | 任意 | 任意 |

**開放問題 Q1**：閾值要老闆確認，特別是冠軍 Frequency ≥6（業界 Big 6 標準）vs 我們的年訪頻率 3.36 偏低，可能要降到 ≥5。

### 4.5 Alert 規則（5 條）

| 警報 | 黃燈閾值 | 紅燈閾值 |
|---|---|---|
| YoY 月營收（trailing 3M） | -10% ~ -15% | <-15% 或連 2 月跌 |
| 新客 90 天回訪率 | <40% | <35% |
| 離店再預約率 | <50% | <30% |
| 客單價（vs 12 月均） | -10% ~ -15% | <-15% |
| 月活躍客戶數 | <70 | 連 2 月 <60 |

排序：紅燈 > 黃燈，最多顯示 3 條。

### 4.6 API 變更

`GET /api/reports?view=daily&date=YYYY-MM-DD`
`GET /api/reports?view=monthly&period=YYYY-MM`
`GET /api/reports?view=annual&period=YYYY`

各 view 回傳結構不同（見下方各 phase 細部）。

## 5. Phase C — 每日視角（HTML 已提供）

### 5.1 對應使用者提供的 daily-dashboard.html

**核心 mental model**：老闆每天工作完 8pm 走進這頁 → 逐筆對帳 → 結班 → 產出 PDF（給帳本）。

#### ① Header — 日期 + 切換器
- 日期 + 星期 + 「營業中 · 11:14 開店」
- 右上角「‹ 4/24」回前一天的歷史對帳

#### ② Hero — 三段總覽
- 今日營收（大字）+ 「↑ 12% vs 4 週週六中位數」
- 服務客數（mid 字）+ 客單
- 對帳進度（mid 字）+ 「還有 N 筆待確認」
- 7 格進度條（綠色 = 已確認 / 灰色 = 待確認）
- 「💡 點擊每筆確認」hint + 「全部確認 →」按鈕

#### ③ 支付方式分類雙卡
- 現金（💵 emoji，小卡）：總額 + 「已確認 X / Y」
- 轉帳（🏦 emoji，小卡）：同上

#### ④ 今日預約對帳清單（核心）
- 篩選器：全部 N / 待確認 N / 異常 N（active 為深底白字）
- 每筆 booking 一行 grid：時間 | 客名+服務 | 金額 | 支付方式 pill | 確認按鈕
- 三種狀態樣式：
  - **confirmed**（已對）：背景 secondary 變淡 + 綠色「✓ 已對」按鈕（不可點）
  - **warning**（異常 / 招待）：黃色細邊框 + 黃色「標記」按鈕
  - **pending**（待確認）：白底 + 黑邊「確認」按鈕
- 「💡 點擊任一筆可編輯金額或支付方式」隱性入口

#### ⑤ 例外操作 3 按鈕 grid
- ➕ 新增 Walk-in
- ⊘ 標記 No-show
- 📝 補登訂單

#### ⑥ 結班按鈕（依進度動態鎖）
- 未完成：disabled「還有 N 筆待確認 · 完成後可結班」+ hint
- 完成：enabled「✓ 結束今天 · 產出對帳單 PDF」+ 「結班後當日資料鎖定 · 補登可從『⋯』進入」

### 5.2 結班 lock 機制（新概念）

「結束今天」按鈕觸發後：
1. 當日所有 booking 被「鎖定」— 一般編輯 UI 不能改
2. PDF 產出當日對帳單（瀏覽器 print to PDF + @media print）
3. 第二天打開 daily 預設看新一天，要看舊資料從 header 「‹ 4/24」navigate

**Schema**：每個 Booking 上加 `settledAt: DateTime?` — `settledAt != null` 表示老闆已對帳。
**結班**：在 `Tenant` 加 `dayClosedAt` map（jsonb），key = ISO date，value = closing time。

### 5.3 互動行為（從 mockup script 對應）

從 `daily-dashboard.html` 的 JS 對應到 React state：
- `confirmRow(btn)` → 單筆 PATCH `/api/bookings/:id/settle` → 更新 `settledAt`
- `confirmAll()` → batch PATCH 多筆
- 篩選器（all / pending / warning）→ 純 client side filter
- 結班按鈕 → POST `/api/admin/day-close` → 寫 `Tenant.dayClosedAt`，前端跳 PDF print dialog

### 5.4 實作檔

**新檔**：`src/app/(admin)/reports/views/daily.tsx`
**新 API**：`PATCH /api/bookings/:id/settle`、`POST /api/admin/day-close`
**新元件（在 Phase A 已列）**：`<BookingRow>`（特殊版，含 confirm 按鈕 3 狀態）

## 14. Phase H — 服務分群自動推播系統（合併原 V3.5 Phase 4-5）

> **取代**：原 V3.5 plan 的 Phase 4 (Coupon 95→9折) + Phase 5 (30 天回訪推播 cron)
> **依據**：4/27 使用者統計分析（去極端值後核心客中位週期）

### 14.1 統計學基礎（給後端工程師理解 why）

**原始平均 vs 中位 vs 核心客中位**：

| 服務 | 原始平均 | 原始中位 | IQR 去極端 | 核心客中位（前 75%） | 行銷推播該用 |
|---|---|---|---|---|---|
| 剪髮 (n=541) | 56 天 | 43 天 | 42 天 | **35 天** | 高信度 ✅ |
| 染髮 (n=25) | 78 天 ⚠️ | 35 天 | 35 天 | **31 天** | 需分群 ⚠️ |
| 燙髮 (n=8) | 140 天 | 146 天 | 146 天 | **122 天** | 業界週期 ⚠️⚠️ |
| 護髮 (n=7) | — | — | — | — | **不發券，產品問題** |

**為什麼染髮平均 78 天但實際 31 天？**
原始 25 筆中宋小姐 / 元榕兩位「死忠客」貢獻 16 筆 30 天左右的記錄，但有 4 筆 200+ 天的「斷檔回流」拉高平均。去除這些極端值後，染髮的真實週期是 31-35 天，跟剪髮差不多。

**統計商業學三原則**：
1. 永遠用中位數，不用平均（被偶爾客污染）
2. 染燙必須分群（死忠 30 天 vs 偶爾 200+ 天，發同樣優惠券對其中一邊一定錯）
3. 三段式：軟提醒 → 優惠券 → 召回券（不要一上來就 9 折，對自己會回的客戶是浪費毛利）

### 14.3 死忠客排除邏輯（做法 B，推薦）

```typescript
function isLoyalCustomer(userId: string, serviceCategory: ServiceCategory): boolean {
  const dates = getServiceHistory(userId, serviceCategory);
  if (dates.length < 3) return false;  // 資料不足

  const intervals: number[] = [];
  for (let i = dates.length - 3; i < dates.length - 1; i++) {
    intervals.push(daysBetween(dates[i], dates[i + 1]));
  }
  const median = calculateMedian(intervals);

  // 中位間隔 < 全店該服務 q25 → 死忠客
  const STORE_Q25 = { 剪髮: 29, 染髮: 30, 燙髮: 97 };
  return median < STORE_Q25[serviceCategory];
}
```

死忠客 → 跳過 軟提醒 + 優惠券（避免送毛利），但**召回券仍發**（流失救援）。

### 14.4 每日 Cron Job

`src/app/api/cron/retention-push/route.ts`，每日 02:00 UTC（10:00 Taipei）：

```typescript
async function dailyPushScheduler() {
  const today = nowTaipei();
  const stages = ["softReminder", "discount10", "winback"] as const;

  for (const service of ["剪髮", "染髮", "燙髮"]) {
    const rules = RETENTION_RULES[service];
    for (const stage of stages) {
      const days = rules[stage];
      const targetDate = subtractDays(today, days);
      const customers = await getCustomersWithLastService(service, targetDate);

      for (const c of customers) {
        // 死忠客排除（軟提醒 + 優惠券，召回券例外）
        if (stage !== "winback" && await isLoyalCustomer(c.id, service)) continue;

        // 7 天 cooldown 防騷擾
        if (await hasReceivedPushInLast7Days(c.id)) continue;

        // 全店每日推播上限
        if (await todayPushCount() >= DAILY_PUSH_LIMIT) break;

        await sendPush(c, service, stage);
      }
    }
  }
}
```
